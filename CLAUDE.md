## HOW（ルールとコマンド）

### 基本ルール
- **必ずファイルを読んでから変更する**（推測で編集しない）
- **変更は最小限に**（頼まれた範囲だけ変更する）
- **破壊的操作（ファイル削除・DB削除など）は必ず確認を取る**

### コミュニケーションルール（毎回必ず守ること）
- **質問は一つずつ行う**（複数の質問を一度に並べない）
- **質問するときは `Q〇. 質問内容（全X問中Y問目）` の形式**で明示する
- 例：`Q1. 既存プロジェクトへの追加ですか？（全3問中1問目）`

### 作業前説明ルール（毎回必ず守ること）
- **どんな作業でも、実施前に日本語で何をするか一言伝えてから動く**
  - 例：「〇〇ファイルを読みます」「△△を修正します」
- **実装・機能追加・新規ファイル作成の場合は、必ず以下の手順を踏む**：
  1. 要件・実装方針をチャットに日本語で提示する
  2. ユーザーの承認を得る
  3. 実装する
  4. 要件をファイルとして `docs/requirements/YYYY-MM-DD_要件名.md` に保存する

### 自動ファイル作成ルール（毎回必ず守ること）
以下のタイミングで、自動的にファイルを作成・更新する：

| タイミング | 作成するファイル | 保存場所 |
|-----------|----------------|---------|
| 実装・機能追加の開始時 | `YYYY-MM-DD_要件名.md` | `docs/requirements/` |
| 会話の中で決定事項が出たとき | `YYYY-MM-DD_決定事項.md` | `docs/decisions/` |
| やることが複数出てきたとき | `YYYY-MM-DD_タスクリスト.md` | `docs/tasks/` |

- ユーザーへの確認は不要、自動的に作成・追記する
- 各プロジェクト内の場合は `Project/app_*/docs/` 配下に保存する

### 会話スタイルルール（毎回必ず守ること）
- **作業の節目ごとに「ここまでで大丈夫ですか？」と確認する**
- **選択肢がある場合は必ず A / B 形式で提示して選んでもらう**
- **作業完了後に「次にやると良さそうなこと」を提案する**
- **迷ったことや判断した理由を都度共有する**
- 丁寧で親しみやすいトーンで話す（ですます調）

### パーミッション・Hooks設定
- ファイル読み書き・編集・通常コマンド（python, npm, git status など） → **自動OK**（設定済み）
- 危険な操作（`rm`, `git push`, `git reset --hard` など） → **Hookが日本語で警告・一時停止**
- 設定ファイル: `.claude/settings.local.json`
- Hookスクリプト: `.claude/hooks/pre-bash-check.sh`

### Python/Flaskプロジェクト共通
- 仮想環境は `venv/` を使用
- 依存パッケージは `requirements.txt` で管理
- テンプレートは `templates_html/` または `templates/`

### TypeScript/Reactプロジェクト共通
- パッケージ管理は `npm`
- ビルドツールは Vite
- スタイルは TailwindCSS

### 新規アプリ作成ルール
- **必ず `app_` から始まるフォルダを作成し、その中で開発する**
- 例: `app_新機能名/`
- フォルダ作成前にユーザーに名前を確認する

### よく使うコマンド
```bash
# Python/Flask
python app.py

# TypeScript/React
npm run dev
npm run build
```

## ログ運用ルール

### ログの種類
| 種類 | 格納場所 | 対象 |
|------|---------|------|
| ワークスペース全体ログ | `logs/YYYY-MM-DD_作業ログ.md` | CLAUDE.md の変更・全体的な作業 |
| プロジェクトログ | `Project/app_*/logs/YYYY-MM-DD_作業ログ.md` | 各プロジェクト内の作業 |

### ログのタイミング（毎回必ず守ること）
- **指示を受けたら即追記**（指示内容をログに記録）
- **ファイルを変更・作成したら即追記**（変更内容をログに記録）
- **重要な判断をしたら即追記**（判断内容と理由をログに記録）
- **セッション終了までまとめて書くのは禁止**（節目ごとにこまめに追記する）

### ログのフォーマット
```markdown
# 作業ログ YYYY-MM-DD

## [HH:MM] 指示
- ユーザーからの指示内容

## [HH:MM] 作業内容
- 実施した作業

## 変更・作成ファイル
| 操作 | ファイル |
|------|---------|
| 新規 / 変更 / 削除 | パス |

## 次回やること
- （あれば）
```
