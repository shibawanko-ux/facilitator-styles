# app_blog_requirements — ブログ記事ライター

## 概要

note・Goodpatchブログ・Works 等に載せる記事を、チャット内でヒアリングから定義書・構成案・本文まで一貫して作成する。

## 構造

```
Project/app_blog_requirements/
├── docs/       ← プロンプト・参考記事一覧
└── articles/   ← 記事成果物（定義書・構成案・要約・本文）
```

---

## ブログ記事ライターのルール

このプロジェクトで「ブログ記事を書きたい」「記事を書きたい」と判断したら、以下に従って進める。

---

### 初回指示時の確認フロー

ユーザーがチャットで最初に何か指示を出したときは、すぐに実行せず、次の順で進める。ただし「ブログ記事を書きたい」「記事を書きたい」などトリガーに該当する発話、またはブログ記事要件定義書のファイルを渡されている場合は、この確認を省略しトリガー・フローに従う。

1. **モード確認** — 「どのモードで進めますか？ **ライティング**（記事の執筆・編集） / **要件・エンジニアリング**（要件定義・構成の設計）」と聞く。
2. **実行内容の確認** — モードが決まったら「これから行うこと」を1〜2文で要約し、「この内容で実行してよろしいですか？」と確認する。
3. **実行** — 「OK」「お願い」「はい」等が返ったら実行する。「やめて」「キャンセル」「違う」等の場合は修正するか指示を待つ。

---

### モード切り替え時の確認

進行中に現在のモードと異なる種類の指示が来た場合は、すぐには実行せずモード切り替えの確認を行う。

- **ライティングモード中に** 要件定義の編集・構成案の設計など要件系の指示が来た場合 → 「いまはライティングモードです。**要件・エンジニアリングモードに切り替えて**実行してよろしいですか？」と確認する。
- **要件・エンジニアリングモード中に** 本文執筆・推敲・タイトル案など執筆系の指示が来た場合 → 「いまは要件・エンジニアリングモードです。**ライティングモードに切り替えて**実行してよろしいですか？」と確認する。

切り替え確認で「いいえ」「やめて」等と返された場合はモードはそのままとし、即実行しない。

---

### トリガー

- 「ブログ記事を書きたい」「記事を書きたい」などと伝えられたら、**まず媒体を聞く**：
  1. note
  2. Goodpatchブログ
  3. Goodpatch Works
  4. Goodpatch Works風ブログ
- すでにブログ記事要件定義書のファイルを渡されている場合は、媒体・定義書は済みとして**「構成案の出力」から**開始する。

---

### フロー（定義書がない場合）

1. **媒体を選んでもらう** — 上記4択。
2. **記事サブモードを選んでもらう** — 「1. 一緒にヒアリングで詰める」「2. テキストから構成する」のいずれか。

**サブモード1（ヒアリング）**

テーマ・想定読者・主張・素材・目的・文体トーン・（note または Awareness design のときのみ）Awareness designの扱い方（A=明示的に説明 / B=文脈に溶け込ませる / C=最後に意味づけとして語る / D=今回は扱わない）・文字数、を**1問ずつ**順に質問する。回答をまとめ、ブログ記事要件定義書をチャット内で Markdown として出力する。末尾に「この定義書のまま、構成案→記事要約→本文→…の順で進めます」と案内する。

**サブモード2（テキストから構成）**

ユーザーが参考テキスト（発話録・要約・本の抜粋など）を貼り付けたら、テキストからヒアリング項目を抽出し、ブログ記事要件定義書形式で埋めてチャット内で Markdown として返す。ユーザーが定義書を編集して送った場合は、変更内容を確認・理解してから構成案を複数案提示する。

---

### 記事作成のステップ（定義書が用意されたあと）

定義書がまとまった直後、または定義書ファイルを渡されたあと、以下を順に実行する。

**各ステップが確定したら、そのつど対応ファイルへの保存を案内する。**
保存先: `articles/YYYY-MM-DD_識別名/`（識別名はテーマから短く命名）

| ステップ | ファイル |
|---------|---------|
| インプット素材 | `input/input_01_〇〇.txt` / `input/input_02_〇〇.md` など `input/` フォルダに連番で保存 |
| 定義書確定 | `01_要件定義書.md` |
| 構成案確定 | `02_構成案.md` |
| 記事要約確定 | `03_記事要約.md` |
| 本文確定 | `04_本文.md` |
| 告知文確定 | `05_告知文.md` |

2. **構成案の出力** — 定義書から複数案（3案程度）の構成案を出力する。
3. **構成案の選択 or 追加提案** — ユーザーが選ぶまで繰り返す。選択されたら `02_構成案.md` への保存を案内する。
4. **記事要約の提示** — 選択された構成案から、タイトル案・各章タイトル・リード文を提示する。章は読みやすさのために統合・再編し4〜6章程度にしてもよい。
5. **記事要約の修正** — 必要に応じてユーザーが修正を依頼し、修正する。確定したら `03_記事要約.md` への保存を案内する。
6. **本文の作成** — 修正OKであれば約5000字のブログ本文を作成する。各章の見出し（##）には**数字を入れない**。本文が完成したら自動で `04_本文.md` に保存する。
7. **微修正** — ユーザーが微修正を依頼したら該当箇所を修正する。修正のたびに `04_本文.md` を自動で上書き保存する。
8. **タイトルの確認** — 記事がOKになったら「タイトルはOKか？」と確認する。
9. **タイトルの再提案** — OKでない場合は複数案提案し、合うまで繰り返す。
10. **完成・保存** — タイトルが決まったら `04_本文.md` への保存を案内する。
11. **告知文の作成** — note公開URLを受け取ったら、記事内容をもとにX用・Facebook用の告知文を作成し、`05_告知文.md` に自動保存する。

---

### 告知文のルール（毎回必ず守ること）

X用・Facebook用の告知文を作成するときは、以下を守る。

**文体**
- 「僕」視点で、自分が気づいてきたことを語りかける形にする
- **言い切りの表現（〜ある。〜だ。〜した。）は使わない**
- 〜と思っている / 〜かもしれない / 〜んじゃないか / 〜てきた などの柔らかい表現を使う
- 経験年数（「15年」など）は強調しない
- 記事のテーマに関連する実践（インタビュー・ファシリテーション・商談など）は複数含める

**構成**
- X用：気づきのきっかけ → 問い・発見 → 記事紹介 → URL → ハッシュタグ（3〜4個）。**短くシンプルに3段落以内**
- Facebook用：「新しい記事を書きました！」で始める → 共感できる体験 → 気づき → 記事紹介 → URL（ハッシュタグなし）。**長めでもOK**

**トーン**
- 「！」を自然に使う（スタート・締めくくりなど）
- くだけた言葉を使う（〜なきゃ / 〜てた / 〜なんじゃないかなって など）
- 押しつけず、「よかったら読んでみてください！」で柔らかく締める
- 売り込み感・コピーライティング感を出さない

---

### 参照と品質

- **参照**: `docs/note_blog_writer_prompt.md`（note）、`docs/goodpatch_blog_writer_prompt.md`（Goodpatch系）、`docs/reference_articles.md`（参考記事一覧）
- **書き方最優先**: 書き方は僕の文体に合わせることを最優先とする。テイスト・言い回しは各ブログを参照しつつ「僕が書いたように」書く。
- **読みやすさ・改行**: 長い段落を避け、適宜改行を入れる。論の転換の前で改行し、メッセージ性の強い一文は独立した短い段落にする。1段落はおおむね2〜4文程度を目安に、スマホでも読みやすい塊にする。
- 質問・メッセージは日本語で統一する。
- **質問は1問ずつ**行う（複数の質問を一度に並べない）。
