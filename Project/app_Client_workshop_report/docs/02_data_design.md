# データ設計：プロジェクト単位の格納

本ドキュメントは [01_requirements.md](01_requirements.md) の「6. データ設計」で参照する、プロジェクト（1レポート）ごとのファイル作成とアウトプット格納の設計である。

---

## 1. 利用フロー（プロジェクト名から始める）

**最初にプロジェクト名を聞く**ところから始める。

1. **プロジェクト名の入力** … 新規のときは「プロジェクト名」（例: `20260130_アイフル`、`20260225_○○社_ResearchProcess`）を入力。既存プロジェクトを続きから編集する場合は一覧から選択する。
2. **プロジェクトフォルダの作成 or 選択** … 入力された名前で `projects/` 配下にフォルダを作成する（新規）、または既存フォルダを選択する。
3. **入力** … そのプロジェクトに紐づけて Good/More・印象が強いメンバー・各ファシリの所感を入力する。必要に応じて `input/` に保存する。
4. **生成** … ①〜⑨の順で編集・生成し、結果を **そのプロジェクトの `output/`** に格納する。

---

## 2. プロジェクトの単位

- **プロジェクト** = 1回のワークショップ振り返りレポート（1クライアント・1回実施ごと）。
- **1プロジェクト = 1フォルダ**で、`projects/` 配下に `プロジェクト名/` として作成する。

---

## 3. フォルダ構造

```
Project/app_Client_workshop_report/
├── docs/                         # 要件定義・プロンプト・サンプル
├── projects/                     # プロジェクト（レポート）格納用
│   ├── README.md                 # 本フォルダの説明・命名ルール
│   └── プロジェクト名/              # 例: 20260130_アイフル
│       ├── meta.json             # 任意：クライアント名・実施日・WS種別など
│       ├── input/                # 入力テキスト（任意で保存する場合）
│       │   ├── good_more.md      # Good/More（チーム別）
│       │   ├── members.md       # 印象が強いメンバー
│       │   └── impressions.md   # 各ファシリの所感
│       └── output/               # 生成アウトプット（①〜⑨）
│           ├── 01_good_more.md
│           ├── 02_members.md
│           ├── 03_impressions.md
│           ├── 04_trends.md
│           ├── 05_hypothesis.md
│           ├── 06_summary.md
│           ├── 07_overview.md
│           ├── 08_background_answers.md   # ⑧の確認回答（⑨を作る場合）
│           └── 09_support_program.md
```

---

## 4. プロジェクトフォルダの命名規則

- **推奨**: `YYYYMMDD_クライアント名` または `YYYYMMDD_クライアント名_WS名`
- 日付＋識別子で、一覧したときに「どのレポートか」が分かる形式とする。
- 例: `20260130_アイフル`、`20260225_○○社_ResearchProcess`

---

## 5. input/ と output/ の役割

| フォルダ | 役割 |
|----------|------|
| **input/** | ファシリテーターが入力した生テキスト（Good/More・印象が強いメンバー・各ファシリの所感）を任意でファイルとして保存する。アプリで入力画面から保存する場合に使用。 |
| **output/** | 編集・生成した ①〜⑨ のセクション別テキストを格納する。スライド（P3, P9〜P15）に貼る文言の出所として参照する。生成のたびに上書きまたは版管理する。 |

---

## 6. output/ 内のファイルとセクションの対応

| ファイル | セクション | スライド上の主な配置 |
|----------|------------|------------------------|
| 01_good_more.md | ① Good＆More（チーム別） | P9〜P12 |
| 02_members.md | ② 印象が強いメンバー（事実＋応援・背中を押す文章） | P13 |
| 03_impressions.md | ③ 各ファシリの所感 | P14 |
| 04_trends.md | ④ 全体傾向 | P3 |
| 05_hypothesis.md | ⑤ 組織課題仮説 | P3 |
| 06_summary.md | ⑥ まとめ（リード文＋要素3点＋総合のGood＆More＋見えた組織課題仮説） | P3 |
| 07_overview.md | ⑦ ワークショップ総括 | P3 |
| 08_background_answers.md | ⑧ 背景確認の回答（イチオシプログラム指定を含む） | （⑨作成時のみ） |
| 09_support_program.md | ⑨ 各種支援プログラム文（全体のタイトル・全体サマリー・傾向と課題仮説・おすすめワークショップ） | P15 |

⑥ の 06_summary.md には次の4ブロックを含める。リード文（150文字程度。全体総評として結論づける。最後に「次のステップが実務につながる」文脈を含める）と要素3点（**今後やったほうがよい3点**を箇条書き。各20文字程度。推奨・次のアクションとして読める形）に加え、**総合のGood＆More**（①を総合した Good 3条・More 3条、各80文字程度）と**見えた組織課題仮説**（全入力から分析した3項目、タイトル＋説明。「〜傾向が見られた」「〜可能性がある」に留める）。

⑨ の 09_support_program.md には**全体のタイトル**（前回体験を起点に次で何を探るかを1文で。例「〜を起点に、〜を体験的に探る」）、全体サマリー、**参加者全体の傾向と課題仮説**（タイトル・箇条書き3点・リード文。リード文は「課題の要約＝次のワークショップを行う理由」とし、プログラム推薦ロジックは書かない）、**おすすめワークショップ**（内容・理由・ねらい）を含める。

---

## 7. meta.json（任意）

プロジェクトのメタ情報を保持する場合の例。実装時に項目を拡張してよい。

```json
{
  "projectName": "20260130_アイフル",
  "clientName": "アイフル株式会社",
  "date": "2026-01-30",
  "workshopType": "Research Process Workshop"
}
```
