# クライアント向けワークショップ振り返り資料作成アプリ 要件定義書

## 1. プロジェクト概要

### 1.1 アプリ名

**クライアント向けワークショップ振り返り資料作成アプリ**（仮）

### 1.2 目的

ワークショップ実施後、ファシリテーターの所感・気づきを「入力テキストのみ」から整理し、対外提出可能な振り返りレポート（スライド想定）を安定して作成する。

### 1.3 背景

- 実施後必ずクライアント向けレポートを作成し、プレゼンする運用のため、随時作成する必要がある。
- GPT 等で編集者役のプロンプトを使っているが、アプリ化して入力〜出力の流れと品質ルールを一元化したい。

### 1.4 主なユーザー・利用シーン

- **主なユーザー**: ワークショップを実施するファシリテーター（レポート作成者・プレゼン担当）。
- **主な読み手**: クライアント（特に事務局）。事務局が実施内容と所見を把握し、納得できる表記とする。
- **利用シーン**: ワークショップ終了後、Good/More/印象に残ったメンバー/所感を入力し、レポート構成（①〜⑨）に沿った資料を生成 → クライアントへプレゼン。

### 1.5 設計の焦点

- **編集者役**: 入力テキスト以外の情報は使わない。推測・補完・一般論の追加禁止。意味と温度感を保った「整理・言い換え」のみ。
- **ハルシネーション防止**: 評価・断定・因果の決めつけ禁止。事実ベース・書き手の言い回しを保持。
- **固定ルールの遵守**: ⑦の総括文言はWS種別ごとに切り替え、⑧の背景確認必須、各セクションの文字数・表現ルールを仕様として明文化する。

### 1.6 文字編集者の役割（LLM の役割定義）

レポートの編集・整形を行う LLM は **「ワークショップ振り返り資料」用の文字編集者** として振る舞う。

| 項目 | 内容 |
|------|------|
| **役割** | ファシリテーターが記述したテキストのみを素材とし、意味や温度感を変えずに整理し、読み手（クライアント）に伝わる形に整える「編集者」。専門家ではなく編集者として振る舞う。 |
| **専門領域** | 編集・コンテンツデザイン。対象読者に合わせた構成・トーン・情報の優先順位づけ。事実を保持したうえでの明確な構成と、読み手の理解・次のアクションにつながる文言。 |
| **品質基準** | 事実（入力テキストの内容・ニュアンス）は変更しない。言い回し・段落構成・見出しで「読みやすさ」と「対外提出可能な品質」を高める。推測は「〜の可能性」「〜傾向が見られた」に留め、断定を増やさない。クライアント（事務局を含む）が読んで、実施内容・所見の出所が分かり、納得感を得られること。 |
| **禁止事項** | 入力以外の情報の使用、推測・補完・一般論の追加、評価・断定・因果の決めつけ、必要以上の分解・要約。改善提案・解決策の追加（More では方向性のみ）。人物評価・性格判断。 |

この役割定義は編集者プロンプト（[10_editor_prompt.md](10_editor_prompt.md)）の冒頭および絶対条件と一致させる。

### 1.7 編集に使用する LLM

- **編集・整形に使用する LLM**: **ChatGPT の最上位バージョン**（OpenAI が提供する中で、現時点で最も能力の高い ChatGPT モデル）を用いる。
- 編集機能は ChatGPT 最上位版（契約・提供状況に応じた最新の最上位モデル）を利用する。実装・運用時にモデル名（例: GPT-4o 等）を追記・更新できるようにする。

### 1.8 役割の付与方法

- 文字編集者の役割（1.6で定義）および絶対条件（プロンプトの制約）は、**編集・整形機能の呼び出し時に、アプリがシステムプロンプト（またはそれに相当する固定プロンプト）として毎回付与する**。
- **ユーザーがチャットで役割を入力・指示する必要はない**。ユーザーが入力するのは、Good / More / 印象が強いメンバー / 各ファシリテーターの所感といった**ソーステキストのみ**とする。

---

## 2. 入力（ソース素材）の定義

アプリが扱う**唯一の素材**として以下を定義する。

- **Good（強み）**: ファシリテーターが記述した強みのテキスト。**チーム（Team A/B/C/D 等）と紐づけて入力・保持する**。
- **More（伸びしろ）**: ファシリテーターが記述した伸びしろのテキスト。**チーム（Team A/B/C/D 等）と紐づけて入力・保持する**。
- **印象が強いメンバーに関する記述**: 理由・行動・姿勢・関わり方（事実ベース）
- **各ファシリテーターの所感**: ファシリテーターごとの所感テキスト（統合・要約しない）

これら以外の情報を生成・補完に使ってはならないことを要件として明記する。

## 3. 出力構成（レポートの9セクション）

編集・生成の**作業順番は、プロンプトの手順（①→②→③→④→⑤→⑥→⑦→⑧→⑨）に従う**。アプリの画面フロー・生成処理の順序もこの順序とする。⑧の背景確認が完了するまで⑨は作成しない。

### 3.1 セクション仕様

- **① ファシリテーターから見た Good & More の整理**  
  Good: 要約タイトル（10〜20文字）＋内容。タイトルは「〜ができていた」「〜が実践されていた」系。リード部分（要約タイトル直下の本文）は、良い点を認めつつ「続けてほしい」「活かしてほしい」など背中を押す文章設計とする。  
  More: 同様に要約タイトル＋内容。タイトルは「〜する」「〜を意識しよう」「〜していく」系。リード部分は伸びしろ・課題として背中を押すトーン（「〜してみてほしい」「〜していくといい」等）とし、硬い語尾の多用を避ける。改善提案・解決策は禁止。  
  いずれもタイトルと本文の重複を避け、本文では状況・背景・次の一歩を補足する。
- **② 印象が強いメンバーの整理**  
  印象に残った理由、行動・姿勢・関わり方（事実ベース）。人物評価・性格判断は禁止。見出しや冒頭で、ファシリテーターの観察に基づく所見であることが分かるようにしてよい。**ワークショップで特に良かった人**であるため、事実に加えて応援や「〜してほしい」など背中を押す文章で締める。硬い「印象的だった」「おられました」の多用は避ける。
- **③ 各ファシリテーターの所感整理**  
  所感は統合しない。ファシリテーターごとに整理。言い回し・温度感を保持。③はファシリテーターから参加者へのメッセージとして呼びかけ調を保持してよい。レポート全体の性格として、ファシリテーターの所見と参加者向けの所感を含むことを前提とする。
- **④ ワークショップ後の全体傾向（必要な場合）**  
  Good/More 各3項目・50文字程度。More は「〜傾向が見られた」を基本表現。
- **⑤ 見えた組織課題仮説（必要な場合）**  
  タイトル20文字以内・リード文50文字以内。最大3項目。「〜可能性がある」「〜傾向が見られた」に留める。**スライド上の配置は P3**（実施WSサマリー内。「見えた組織課題仮説」ブロック。タイトル＋箇条書き3項目の形式）。
- **⑥ まとめ**  
  **リード文**（150文字程度）: 今回のインプット内容（①〜③および総合のGood＆More・見えた組織課題仮説）をすべて踏まえ、全体総評として結論づける文章にする。最後の一文で「次のステップが実務につながる」文脈を必ず含める。  
  **要素3点**: レポートの構成を示す名詞句ではなく、**今後こういうことをやったほうがよい3点**を箇条書きで記載する（推奨・次のアクションとして読める形）。各項目は**20文字程度**とする。  
  さらに **総合のGood＆More**（①を総合した Good 3条・More 3条、各80文字程度）と **見えた組織課題仮説**（全入力から分析した3項目、タイトル＋説明。「〜傾向が見られた」「〜可能性がある」に留める）を P3 用に 06_summary.md に含める。
- **⑦ 今回のワークショップ総括（必須）**  
  **WS種別ごとに総括文言を切り替える**。マスタまたは設定で「実施ワークショップ種別」に対応した文言を保持する（例: Research Process Workshop の場合は「今回のワークショップ総括：リサーチを深める「問いと抽象化の力」」）。ユーザーによる任意の変更は不可とするか、デフォルトで固定とする。
- **⑧ 各種支援プログラム文の背景確認（必須フロー）**  
  支援プログラム文を作る前に、次の4項目をユーザーに確認する: (1) なぜこの支援プログラムを選んだか (2) 直前に実施したWS・体験 (3) 参加者にどんな続きの体験をしてほしいか (4) 現時点で気になっている点・引っかかり。回答が揃うまで⑨を作成しない。**イチオシとして推薦するプログラム名**（例: インタビュー Workshop）を、⑧の入力時または回答に含めて指定できるようにする。
- **⑨ 各種支援プログラム文**  
  **全体のタイトル**（1文。前回の体験を起点に、次のプログラムで何を体験的に探るかを示す。例:「〜を起点に、〜を体験的に探る」形式）。全体サマリー文（約80文字）。**参加者全体の傾向と課題仮説**（タイトル約30文字・箇条書き3点・**リード文約80文字**）。**リード文は「課題の要約＝次のワークショップを行う理由」**として記述し、特定プログラムの推薦ロジックは書かない。推薦の理由・ねらいは「おすすめワークショップ」ブロック（内容・理由約100文字・ねらい約100文字）に書く。新しい価値提案・営業的表現は禁止。**⑨に限り、ワークショップメニュー（プログラム一覧・説明）を参照してよい**（絶対条件「入力テキスト以外使用しない」の例外）。

### 3.2 セクションとスライドの対応

文言をアプリで作成するスライドは **P3** と **P9〜P15** のみ。それ以外は手作業で編集する。

| セクション | スライド | 備考 |
|------------|----------|------|
| ⑦ 総括、④ 全体傾向、⑥ まとめ、⑤ 組織課題仮説 | P3 | 実施WSサマリー。⑤は「見えた組織課題仮説」ブロック（タイトル＋箇条書き3項目） |
| ① Good＆More（チーム別） | P9〜P12 | チーム数に応じて複数ページ |
| ② 印象が強いメンバー | P13 | 推薦理由・行動・関わり方 |
| ③ 各ファシリテーターの所感 | P14 | メイン／サブごと |
| ⑨ 各種支援プログラム文 | P15 | 傾向と課題仮説、イチオシ推薦 |

表紙・Agenda・参加者（座席表）・Appendix は手作業で編集する。

### 3.3 出力の想定形（最終サポートのスライド構成）

- 生成した文章は、**スライド資料の各枠にコピー・配置して完成させる**運用とする。
- **参照サンプル**: `docs/samples/20260130_アイフル株式会社御中_ワークショップ振り返り.key.pdf` を最終サポートのサンプルとして参照する。
- **文言を作成する場所**: **P3** と **P9〜P15** のみ。それ以外（表紙、Agenda、参加者/座席表、Appendix）は手作業。
- **チーム単位の Good/More**: 入力・出力はチーム別を前提とし、入力UIではチーム名またはチームIDと紐づける。役割表記はメインファシリテーター、サブファシリテーター（B:サブ、C:サブ、D:サブ 等）をサンプルに合わせる。

### 3.4 各種支援プログラム文とワークショップメニュー

- ⑨で推薦するプログラムは、**ワークショップメニュー一覧に掲載されているプログラム**に基づく。参照メニュー: `docs/samples/Workshopメニュー.key.pdf`（またはプロジェクト内にコピーしたパス）。メニューの分類（イチオシ・おすすめ・候補）およびプログラム名・概要等は、メニューPDFまたはデータ化したマスタに依存する。⑧の背景確認が取れたうえで、今回の振り返りから自然につながるプログラムをメニューから選び⑨の文言とする。新しい価値提案・営業的表現は禁止。

## 4. 機能要件

### F-0 手順アテンド（ガイドモード）

**チャットで「始めましょう」と入力した場合**、手順（[00_手順.md](00_手順.md)）に従って**一つずつアテンド**しながら進める。

- 各ステップで「次に何をするか」を案内する。
- ユーザーの入力や確認を待ってから、次のステップに進む。
- 手順の 1（プロジェクトを決める）→ 2（入力する）→ 3（①〜⑦を生成）→ 4（⑨を作る場合）→ 5（スライドに貼る）→ 6（手作業で仕上げる）の流れに沿って、順にアテンドする。

利用フローは**最初にプロジェクト名を入力（または既存プロジェクトを選択）**するところから始める（F-0 のアテンドでは、その案内がステップ1として行われる）。詳細は [02_data_design.md](02_data_design.md) を参照。

### F-1 入力機能

上記4種のソース（Good / More / 印象が強いメンバー / 各ファシリテーターの所感）を入力・編集できる。**入力は一つずつ行い、送信のたびに結果を出力する**（一括入力ではなくステップごとに入力する）。

- **チームごとに Good と More** … Aチーム → 送信・結果出力。Bチーム → 送信・結果出力。以降、チームがなくなるまで繰り返す。ユーザーが「次へ」と言ったら次の入力ステップへ進む。
- **印象が強いメンバー** … 「次へ」のあと、誰（名前）と内容を箇条書きで入力 → 送信・結果出力。「次へ」で次へ。
- **各ファシリテーターの所感** … 「次へ」のあと、名前と内容を箇条書きで入力 → 送信・結果出力。ここまでで入力完了とし、続いて ①〜⑦の生成に進む。

Good・More はチーム（Team A/B/C/D 等）と紐づけて保持する。ファシリテーターの所感は「1人1ブロック」で保持する。手順の詳細は [00_手順.md](00_手順.md) の「2. 入力する」を参照。

### F-2 編集・整形機能

入力テキストに基づき、①〜⑦（および必要に応じて④⑤）のセクションを生成する。

- **作業順番**: 編集・生成は**プロンプトの手順どおり ①→②→③→④→⑤→⑥→⑦ の順**で行う。⑧（背景確認）を経たのち ⑨ を作成する。
- **使用 LLM**: 編集・整形には **ChatGPT の最上位バージョン**（現時点で提供されている最も能力の高いモデル）を使用する。
- **役割の付与**: 文字編集者の役割および絶対条件は、API 呼び出し時にアプリがシステムプロンプト（または同等）で毎回付与する。ユーザーによる役割の入力は不要とする。
- 実装方針: 上記 LLM と「文字編集者」役割（1.6）および編集者プロンプト（[10_editor_prompt.md](10_editor_prompt.md)）を同一ソースとして参照し、プロンプトは要件から一意に導けるようにする。手動編集のみの運用でも、画面・導線がこの構成と作業順序に沿うようにする。

### F-3 ⑧ 背景確認フロー

**⑨を作成する場合のみ⑧を表示する**。支援プログラム文を作らない場合は⑧・⑨をスキップする。⑨を作成するときは、⑧の4項目を画面で確認・入力させ、4項目が揃うまで⑨の生成（表示）を無効にする。**イチオシとして推薦するプログラム名**（例: インタビュー Workshop）の入力も受け付け、指定があれば⑨の「おすすめワークショップ」に反映する。

### F-4 総括文言の管理

⑦の総括は**WS種別ごと**にアプリ内のマスタまたは設定で保持する。ユーザーが任意に変更できないようにする（または変更不可をデフォルトとする）。

### F-5 出力・エクスポート

レポート全体をスライド資料向けにエクスポートできるようにする（例: Markdown / PDF / スライド形式のいずれか。優先度は別途決定）。

### F-6 随時作成対応

1ワークショップ = 1レポートとして、過去レポート一覧や新規作成の起動ができる（履歴管理の有無はスコープで調整）。

## 5. 品質要件・コンテンツルール（絶対条件の明文化）

プロンプトの「絶対条件」を要件定義の一文として記載する。

- 入力テキスト以外の情報は使用しない。
- 推測・補完・一般論の追加は禁止。
- 評価・断定・因果関係の決めつけは禁止。
- 意味を変えない範囲での「整理・言い換え」のみ許可。
- 書き手の言い回し・感情の温度感は極力保持する。
- 必要以上に分解・要約しない。
- 専門家ではなく「編集者」として振る舞う。

**例外**: **⑨ 各種支援プログラム文**を作成する際に限り、**メニュー（プログラム一覧・説明）の参照を許容する**。上記「入力テキスト以外の情報は使用しない」は⑨についてはメニュー情報を除く。

**表記ルール**: 略語「ファシリ」を使う場合は、文脈で意味が通じる範囲で使用する。必要に応じて初出時は「ファシリテーション」「ファシリテーター」とし、同一文書内で表記を統一する。

**所見の注釈**: ⑥の総合のGood＆More・見えた組織課題仮説は、ファシリテーターの観察に基づく所見であることを、読み手が理解できるよう必要に応じて短い注釈で明示してよい。

**リード文の品質**: ① Good＆More の各リード部分（要約タイトル直下の本文）は、読み手（事務局）向けに対外提出可能な品質とし、背中を押す文章設計とする。Good は良い点を認め、励ましや「この調子で続けてほしい」「活かしていってほしい」で締める。More は伸びしろ・課題として「〜してみてほしい」「〜していくといい」等、励ましのトーンで締める。硬く聞こえる「〜と見られた」「印象的だった」「うかがえた」の多用は避け、伝わりやすく寄り添う表現にする。タイトルと本文が同語で重複しないよう、本文では状況・背景・次の一歩を補足する。

これらは、LLM を使う場合はプロンプトに必須として含め、手動編集の場合はチェックリストやガイドラインとして画面・ドキュメントに反映する。

## 6. データ設計・非機能・セキュリティ（方針）

- **データ設計**: 利用フローは**最初にプロジェクト名を入力（または既存プロジェクトを選択）**し、そのプロジェクト配下で入力・生成・アウトプット格納を行う。プロジェクト（1レポート）ごとに `projects/プロジェクト名/` を作成し、`input/`（任意）と `output/`（①〜⑨の生成文）を格納する。詳細は [02_data_design.md](02_data_design.md) を参照。MVP ではローカル/ファイルベースでよいかはスコープで決定。
- **非機能**: 随時作成に対応するため、複数ワークショップ分のレポートを扱えること。レスポンス時間・同時利用数は初期は厳格にしない想定で記載。
- **セキュリティ**: クライアント向け資料のため、入力・生成レポートの取り扱い（保存場所・アクセス制御・API キー管理）を簡潔に記載。必要に応じて参照 doc を後から追加する。
