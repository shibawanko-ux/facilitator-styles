# 分析結果画面の拡張要件（検討案）

本ドキュメントは、分析結果画面に「レーダーチャート」「分析結果文章」「振り返り用の問い」「アクション提案」を追加する際の**実施可否の結論**と**要件定義**です。01_requirements.md の 2.4 結果の表示 を拡張する案として位置づけます。

---

## 0. 表示の構成（改訂・優先）

結果画面の**表示順と各ブロックの役割**は **[14_report_structure.md](14_report_structure.md)** に従う。同ドキュメントで「結果画面の全体構成（読み手の流れ）」として、ブロック0〜3・見出し案・読み手の目的を定義している。

- **ブロック0**: 六角形レーダーチャート1枚。6軸、メイン・サブ・参加者の系列。軸ラベル・点数・凡例の配置要件は本ドキュメントの 2.3（R-1〜R-4）を参照。
- **ブロック1〜3**: レポート部分（あなたへの評価／6つの観点の解説／振り返りの問い・シェアドリーダーシップ）。分量・文言方針・表示形式は 14_report_structure.md を参照。

**デザインの方向性**は [15_design_direction.md](15_design_direction.md) に従い、ファシリテータースタイル診断と同一ブランドで統一する。

---

## 1. 実施可否の結論

**実施可能です。** 現行のデータ構造（evaluations の role / scores）と集計ロジック（役割別・項目別平均）を拡張すれば実現できます。以下、3つの要望ごとに技術的な前提と要件を整理します。

---

## 2. レーダーチャートの追加

### 2.1 要望の整理

- **6つの観点**（6コアスキル対応）をレーダーチャートで可視化したい
- **メインファシリテーター**・**参加者**・**サブファシリテーター**の評価の**差分**が1枚で見られる状態にしたい
- **1枚の六角形レーダー**に6つの観点を描き、役割間の差が分かるようにする

### 2.2 実施可否と前提

- **実施可能**
  - 現行で役割（main / sub / participant）ごとに評価が保存されているため、**各軸の総合点**（各軸に属する項目IDの平均。03の6軸対応表に準拠）を役割別に算出し、**1枚の6軸レーダー（六角形）**上に**メイン・参加者・サブ**の3系列（または存在する役割のみ）として表示できる
- **技術前提**
  - フロントでレーダーチャートを描画するライブラリを利用する（例：Recharts の RadarChart。軸6本で六角形になる）
  - resultAnalysis に「**役割別・6軸の総合点**」を返す関数を用意する（例：`computeSectionAveragesByRole(evaluations)` → 6軸それぞれ1値ずつ × main / sub / participant）

### 2.3 要件定義（1枚の六角形レーダー）

| ID | 要件 | 受入条件 |
|----|------|----------|
| R-1 | **1枚の六角形レーダーチャート**を表示する。軸は**6本**（説明・設計、見える化・編集、場の観察、場のホールド・安心感、問いかけ・リフレーミング、流れ・即興）。各軸の値は、その軸に属する項目の**総合点**（03の6軸対応表に従い、該当項目IDの平均）とする | 六角形のレーダーが1枚表示され、6頂点が上記6つの観点に対応している |
| R-2 | 同一レーダー上に**メイン**・**参加者**・**サブ**の**3系列**（または2系列）を重ねて表示し、役割間の差分が視覚的に分かるようにする | 凡例で「メイン」「参加者」「サブ」を区別でき、同一軸上で役割間の差が比較できる |
| R-3 | 参加者のみ・サブのみがいる場合は、**存在する役割のみ**を系列として表示する（例：メイン＋サブのみのときは2系列） | データがある役割だけがチャートに現れる |
| R-4 | 各軸で**観点名を軸の名目位置**に表示し、**メイン・サブ・参加者の点数**は軸の**外向き**にオフセットして配置し、チャート図や他ラベルと重ならないようにする。点数の文字色はチャートの系列色と同一とする。**凡例**（メイン・サブ・参加者の色の説明）はチャート外（下段）に配置し、文字と重ならないようにする | 例：軸に「説明・設計」を名目位置に表示し、その外向きにメイン：3.0、サブ：3.0、参加者：2.0。凡例はチャート下に独立して表示される |

**視点の数**: **6軸（六角形）**とする。

## 3. 分析結果文章の追加

### 3.1 要望の整理

- 6軸それぞれについて、**平均点だけではなく「分析結果のコメント文章」**を表示したい
- 文章は**プロの編集者**が作成し、**読み手（メインファシリテーター）**が理解しやすく、**次のアクションが分かる**ようにしたい

### 3.2 実施可否と前提

- **実施可能**
  - 表示内容は「テキストの出し分け」であり、スコア帯（例：平均 1〜2 / 3 / 4〜5）やセクション平均に応じて、あらかじめ用意した文言を表示する方式で実装できる
  - 文言はデータ（JSON やマスタ）で保持し、**編集者が執筆・更新できる形**にしておく（アプリ外で編集し、アプリは読み込むだけ）
- **前提**
  - 「プロの編集者が書く」部分は**コンテンツ制作（人的作業）**。アプリ側は**どの文章を表示するかのルール**と**文章を格納・参照する仕組み**を用意する

### 3.3 要件定義

| ID | 要件 | 受入条件 |
|----|------|----------|
| A-1 | 6軸**それぞれ**について、**総合点数**と**分析結果のコメント文章**を**箇条書き**で表示する | 各軸のブロックに、総合点（その軸の項目群の平均）と、読み物としての分析文が箇条書きで表示される |
| A-2 | 文章の内容は**メインファシリテーターが理解しやすく、次のアクション（実践可能性）が分かる**ことを目的とする | 要件 01 の 1.2（目的・感情の動きと実践可能性に焦点）と整合。文言の品質は編集者による執筆・レビューで担保する |
| A-3 | 表示する文章は**編集者が編集可能**な形で管理する（例：リポジトリ内の JSON/ Markdown、または CMS。初版は JSON で十分） | 文言の差し替えがコード変更なし、または最小限でできる |
| A-4 | （推奨）表示する文章を**スコア帯や傾向で出し分け**する（例：各軸のセクション平均 4以上／3台／3未満で別文言）。出し分けルールと文言の対応表を定義する | 振り返りの質を高めるため推奨。初版は「1パターン1文言」でも可 |

### 3.4 データ・運用イメージ

- **格納**: 例として reportAgent の sectionCommentTemplates 等、6軸（design, visibility, observation, hold, questioning, flow）に対応した形で保持
- **編集フロー**: 編集者が body を書き、PR またはファイル更新で反映。将来的に CMS や API に移すことも可能

### 3.5 分析結果文章の内容ルール（差分から見る）

分析結果の文章は**差分から見る分析結果文章**として書く。すなわち、メイン・サブ・参加者の総合点の差分（誰が高めか、差が大きいか・近いか）を手がかりに、「その差分から何が見えるか」を述べる内容とする。概念の一般的な説明ではなく、**このルームの点差から読み取れる解釈**を表示する。

| ID | 要件 | 受入条件 |
|----|------|----------|
| R-COM-1 | 分析結果文章の本文に**評価項目ID（E1–E4, B1–B10, F1–F4 等）を書かない** | 結果画面では項目IDを表示していないため、読み手が場所を特定できない。必要なら「全体の設計」「問いかけ」など観点・意味で表現する |
| R-COM-2 | 分析結果は**差分から見る分析結果文章**とする | メイン・サブ・参加者の総合点の差分を前提に、「その差分から見ると何が起きていると考えられるか」を記述した文章を表示する |
| R-COM-3 | 総評の先頭に**事実の1文**を表示する | その目のメイン・サブ・参加者の平均点と、誰と比べてどうか（メインが約○点高め／低め／ほぼ同じ）を1文で表示し、読み手が「何の数値の話か」を把握できるようにする |

**実装方針**: メイン・サブ・参加者の大小関係でパターン（例：メインが他より高い／メインが他より低い／役割間で近い）を定義し、パターンごとに「差分から見る」分析文を1本用意する。表示時に実際の点差からパターンを判定し、対応する文章を表示する。文言は編集者が管理可能な形で保持する。**事実の1文**は、表示時にその目のスコアと役割の有無（roleCounts）から「メイン X.X 点、サブ Y.Y 点で、メインが約Z点高めです」のように組み立てる関数（例：getAnalysisFactSentence）で生成し、総評ブロックの先頭に表示する。

## 4. 振り返り用の問いの設計と表示

### 4.1 要望の整理

- **メインファシリテーター**と**参加者・サブファシリテーター**の**対話のきっかけ**となる問いを表示したい
- 3つの目からの分析結果に基づき、**メインの成長**につながる問いとし、**対話を促す**形にしたい
- 問いは**3つ程度**表記し、**意図**も含めて表現する
- **約1時間**の振り返り時間を想定する

### 4.2 実施可否と前提

- **実施可能**
  - 現行でも「振り返りの問い」「アクション提案」を表示している。これを拡張し、「対話のきっかけとなる3問＋意図」を**固定またはパターン別**で表示する
  - 問いと意図は編集者が執筆し、データ（JSON 等）で保持する

### 4.3 要件定義

| ID | 要件 | 受入条件 |
|----|------|----------|
| Q-1 | 結果画面に**振り返り用の問い**を**3つ程度**、**箇条書き**で表示する | 3問（または2〜4問の範囲で運用で調整）が、対話のきっかけとして箇条書きのブロックで表示される |
| Q-2 | 各問いに**意図**を併記する（なぜその問いをするか、何を引き出したいか） | 問いの下または横に、意図が短い文で表示される |
| Q-3 | 問いは**メインファシリテーターの成長**と、**メインと参加者・サブとの対話**を促す内容とする | 文言の品質は編集者で担保。アプリは「問い＋意図」を表示する枠を用意する |
| Q-4 | 振り返り時間の想定は**約1時間**とし、問いの設計・文言はその時間で扱いやすいようにする | 運用・ファシリテーション側の設計。アプリは「約1時間の振り返り用」と説明できるラベルや注意書きを付けてもよい |
| Q-5 | （推奨）分析結果（スコア帯・傾向）に応じて**出し分ける問い**を用意する（例：流れ・即興が低いときは時間配分に関する問いを優先） | 初版は全ルーム共通の3問＋意図でも可。拡張でパターン別に対応 |

### 4.4 アクションの提案の表示

結果画面の表示構成（0. 表示の構成）に従い、**アクションの提案**ブロックは次の形式で表示する。

| ID | 要件 | 受入条件 |
|----|------|----------|
| AC-1 | **内容の説明文章**を表示する（なぜこのアクションが提案されるか、振り返りの文脈でどう使うかなど） | ブロック先頭に、編集者が執筆した説明文が表示される |
| AC-2 | **箇条書きでアクションを3点程度**表示する | 実行可能な一歩として、3点前後のアクションが箇条書きで表示される |
| AC-3 | 表示内容は**編集者が編集可能**な形で管理する（既存の reflectionByType の action を拡張するか、別データで「説明文＋箇条書き3点」を保持） | 文言の差し替えがコード変更なし、または最小限でできる |

**データ構造の拡張案**: 既存の `ReflectionContent` を `{ question, actionSummary?: string, actionBullets?: string[] }` のように拡張し、`actionSummary` が説明文、`actionBullets` が箇条書き3点とする。未設定の場合は従来の `action` を説明文として表示してもよい。

### 4.5 表示位置・既存との関係

- 表示順は **0. 表示の構成** に従う（レーダー → 総合点数・分析結果 → 振り返り用の問い → アクションの提案）
- 現行の「振り返りの問い」「アクション提案」（メインファシリタイプ別）は、上記構成の**振り返り用の問い**および**アクションの提案**として表示する（内容は 4.3・4.4 の形式に合わせて拡張する）
- 編集者が書くコンテンツは、10_reference_facilitation.md の運用方針に従い、**文言のバリエーションやパターン**を 08 に追記するか、別データ（analysisComments と同様）で管理する

## 5. まとめ・依存関係

| 項目 | 実施可否 | 主な依存 |
|------|----------|----------|
| **1枚の六角形レーダー**（6軸、メイン・参加者・サブの3系列） | 実施可能 | 役割別・6軸の総合点を返す関数、1枚の RadarChart（6軸）で3系列表示 |
| 総合点数＋分析結果文章（6軸を箇条書き） | 実施可能 | 編集者による文言作成、出し分けルールとデータ構造の定義 |
| 振り返り用の問い（箇条書き、3問＋意図、約1時間想定） | 実施可能 | 編集者による問い・意図の作成、表示ブロックの追加 |
| アクションの提案（説明文＋箇条書き3点程度） | 実施可能 | ReflectionContent の拡張（actionSummary / actionBullets）、編集者による文言 |

- **表示順（必須）**: 上から (1) 六角形レーダー → (2) 総合点数・分析結果 → (3) 振り返り用の問い → (4) アクションの提案
- **実装順の提案**: (1) 役割別・6軸の総合点の集計と1枚の六角形レーダー表示 → (2) 総合点数・分析結果の箇条書き表示（6軸）→ (3) 振り返り用の問い・アクション提案の形式統一
- **コンテンツ**: 分析コメント・問い・意図・アクション文は**プロの編集者**が執筆する。アプリは「表示ルール」と「格納形式」を決め、編集しやすい形で持つ。

## 6. 実装メモ（現状）

- **レーダー**: 1枚の六角形レーダー（6軸）。resultAnalysis の `computeSixAxisAveragesByRole` で役割別・6軸の総合点を算出。ResultPage の RadarChartSection で描画。
- **表示順**: 上から レーダー → レポート（セクション1〜3）。詳細は [14_report_structure.md](14_report_structure.md)。
- **分析結果**: reportAgent が6軸の sectionComments を生成。総評の事実文は点数を出さずテキストで表現（renderTemplateComment の buildFactSentence）。
