# ファシリテーターリフレクション 実装ガイド（初心者向け）

このドキュメントは、専門用語をなるべく解説しながら、一緒に作業するように進めるための手順書です。一歩ずつ進めます。

---

## 用語集（よく出てくる言葉）

| 用語 | 意味（やさしく） |
|------|------------------|
| **Supabase** | データを保存したり、あとから取り出したりするための「クラウド上のサービス」。自分でサーバーを用意しなくても、ネット上にデータの置き場所を作れます。 |
| **テーブル** | データを「表」の形で保存する場所。例：ルーム一覧用の表（rooms）、評価一覧用の表（evaluations）。 |
| **カラム** | 表の「列」のこと。例：ルーム名、パスコード、作成日時など、1つ1つの項目がカラムです。 |
| **RLS（Row Level Security）** | 「この人にはこの行（データ）だけ見せる」というルール。ルームごとに鍵をかけて、他のルームのデータが見えないようにするために使います。 |
| **パスコードのハッシュ** | パスコードをそのまま保存せず、不可逆に変換した値で保存すること。万が一データが漏れても、元のパスコードは復元できないようにするためです。 |
| **API** | アプリから「データを保存して」「データを取ってきて」とお願いするための窓口。Supabase が自動で用意してくれます。 |
| **環境変数** | パスワードや接続先のURLなど、コードに直接書かずに外から渡す設定値。`.env` というファイルに書いておき、Git には上げません。 |
| **Vite** | フロント（画面側）のアプリをビルドしたり、開発用のサーバーを起動したりするツール。React と組み合わせて使います。 |

---

## 進め方のイメージ

1. **Step 1** から順に進めます。
2. 各 Step の最後に「ここまでで〇〇ができている」と書いてあります。できたら次の Step へ。
3. わからないところはその Step で質問してから先に進んで大丈夫です。

## Step 1: Supabase のアカウントを作る

**目的**: データを保存する場所（Supabase）を使うために、まずは会員登録をします。

**手順**:
1. ブラウザで https://supabase.com を開く
2. 右上の **Sign in** をクリック
3. **Sign in with GitHub** を選ぶ（GitHub アカウントがある場合。ない場合は **Sign up with Email** でメールアドレスで登録）
4. 画面の指示に従ってログインまたは新規登録する

**ここまでで**: Supabase のダッシュボード（管理画面）に入れる状態になります。

---

## Step 2: 新しいプロジェクトを作る

**目的**: このアプリ専用の「データの置き場」を 1 つ作ります。

**手順**:
1. ダッシュボードで **New Project** をクリック
2. **Name**: 例として `facilitator-reflection` と入力（好きな名前でOK）
3. **Database Password**: データベース用のパスワードを自分で決めて入力（**必ずメモしておく**。あとで使います）
4. **Region**: 日本に近い **Northeast Asia (Seoul)** や **Southeast Asia (Singapore)** を選ぶと通信が速くなりやすい
5. **Create new project** をクリック
6. 1〜2 分待つと「Project is ready」と出ます

**ここまでで**: このアプリ用の Supabase プロジェクトが 1 つできています。

---

## Step 3: 接続に必要な情報をメモする

**目的**: あとでフロントのアプリから「この Supabase に接続する」と指定するときに使う URL とキーを控えます。

**手順**:
1. 左メニューの **Project Settings**（歯車のアイコン）をクリック
2. 左の **API** をクリック
3. 次の 2 つをメモ（またはコピー）する：
   - **Project URL**（例: `https://xxxxx.supabase.co`）
   - **anon public** のキー（**Project API keys** のところにある長い文字列。`eyJ...` で始まります）

**注意**: **service_role** のキーは秘密なので、人に教えたりコードに書いたりしないでください。フロントでは **anon** だけ使います。

**ここまでで**: フロントから Supabase に接続するときに使う「住所」と「鍵」が手元にあります。

---

## Step 4: テーブル（rooms）を作る

**目的**: ルームの情報（ルーム名、パスコード、メインファシリのタイプなど）を保存する「表」を作ります。

**手順**:
1. 左メニューの **Table Editor** をクリック
2. **Create a new table** をクリック
3. **Name**: `rooms` と入力
4. **Columns** で、以下のカラムを 1 つずつ追加する（**Add column** で追加）:

| カラム名 | Type | 設定 |
|----------|------|------|
| id | uuid | 主キー（Primary key）、デフォルト: `gen_random_uuid()` |
| name | text | 必須（Required）にチェック |
| passcode_hash | text | 必須 |
| main_facilitator_type_id | text | 必須 |
| event_date | date | 任意（チェックなし） |
| event_name | text | 任意 |
| created_at | timestamptz | デフォルト: `now()` |

5. **Save** で保存

**ここまでで**: ルーム用の表が 1 つできています。

---

## Step 5: テーブル（evaluations）を作る

**目的**: 誰がどのルームに、どんな評価（スコアやコメント）を送ったかを保存する「表」を作ります。

**手順**:
1. また **Create a new table**
2. **Name**: `evaluations`
3. 以下のカラムを追加:

| カラム名 | Type | 設定 |
|----------|------|------|
| id | uuid | 主キー、デフォルト: `gen_random_uuid()` |
| room_id | uuid | 必須。のちに rooms(id) への参照（Foreign key）にする |
| role | text | 必須（main / sub / participant のいずれか） |
| evaluator_label | text | 任意 |
| scores | jsonb | 必須 |
| free_comment | text | 任意 |
| created_at | timestamptz | デフォルト: `now()` |

4. **Save**
5. あとから **room_id** に、rooms の id を参照する外部キーを設定すると、データのつながりがはっきりします（Step 5 のあとでも、Table Editor から編集できます）

**ここまでで**: 評価用の表もできています。

## Step 7: フロント用のプロジェクト（Vite + React）をパソコンに作る

ここでは、画面（フロントエンド）を動かすための「プロジェクト」を、あなたのパソコン上の `Project/app_Facilitator_reflection` フォルダに用意します。**Vite** と **React**、**TypeScript**、**Tailwind CSS** を使う構成にします（診断アプリと同じです）。

### 用語（このステップで出てくるもの）

| 用語 | 説明 |
|------|------|
| **Vite** | フロントのビルドツール。開発サーバーや本番用のファイル生成を担当します。 |
| **React** | UI をコンポーネント単位で書くためのライブラリ。 |
| **TypeScript** | JavaScript に型を付けた言語。ミスを減らしやすくします。 |
| **Tailwind CSS** | クラス名でスタイルを指定する CSS フレームワーク。 |
| **npm** | Node のパッケージ管理。`package.json` に書かれた依存関係をインストールします。 |

### 手順（すでに同じフォルダに作成済みの場合は「確認」だけ）

1. **Node.js が入っているか確認**
   - ターミナルで `node -v` と `npm -v` を実行し、バージョンが表示されれば OK です。入っていない場合は [Node.js の公式サイト](https://nodejs.org/) から LTS 版をインストールしてください。

2. **プロジェクトの作成（未作成の場合）**
   - `Project/app_Facilitator_reflection` フォルダに、Vite の React + TypeScript テンプレートでプロジェクトを作成します。
   - 既存の `.cursor` や `docs` を残したいため、一度別フォルダ（例: `_vite_temp`）に作成し、中身を移動する方法でも構いません。
   - コマンド例（別フォルダに作成してから移動する場合）:
     ```bash
     cd Project/app_Facilitator_reflection
     npm create vite@latest _vite_temp -- --template react-ts
     # 中身を移動して _vite_temp を削除
     ```

3. **Tailwind CSS の追加**
   - 同じフォルダで次を実行します。
     ```bash
     npm install -D tailwindcss postcss autoprefixer
     ```
   - `tailwind.config.js` と `postcss.config.js` を用意し、`src/index.css` の先頭に `@tailwind base;` など Tailwind の指示を書きます（本プロジェクトではすでに設定済みです）。

4. **依存関係のインストール**
   - 初回や `package.json` を変えたあとは、次を実行します。
     ```bash
     npm install
     ```

5. **開発サーバーの起動**
   - 次を実行すると、ブラウザで開ける URL（例: `http://localhost:5173`）が表示されます。
     ```bash
     npm run dev
     ```
   - ブラウザで「ファシリテーターリフレクション」などの表示が出れば、Step 7 は完了です。

**ここまでで**: フロント用のプロジェクトがパソコンにあり、開発サーバーで画面を表示できる状態になっています。

> **動作確認の起動手順だけ見たい場合** → [09_startup_guide.md](09_startup_guide.md) にまとめてあります。

## Step 8: 環境変数で Supabase の URL と anon キーを渡す

フロントから Supabase（データベース）に安全に接続するため、**環境変数**でプロジェクトの URL と **anon キー**を渡します。秘密情報をコードに書かず、`.env` ファイルで管理します。

### 用語（このステップで出てくるもの）

| 用語 | 説明 |
|------|------|
| **環境変数** | OS やアプリが参照する「名前＝値」の設定。API キーなどをコードに直書きせず渡すために使います。 |
| **anon キー** | Supabase の「匿名用」公開キー。RLS で制限された範囲だけ読み書きでき、フロントで使ってよいキーです。 |
| **.env** | プロジェクト直下に置くファイル。`VITE_` で始まる変数が Vite 経由でフロントに渡されます。リポジトリにコミットしないようにします。 |

### 手順

1. **Supabase の URL と anon キーを確認**
   - Supabase ダッシュボードで、対象プロジェクトを開く
   - **Project Settings**（歯車アイコン）→ **API** を開く
   - **Project URL** と **Project API keys** のうち **anon public** の値をコピーする

2. **.env ファイルを用意**
   - `Project/app_Facilitator_reflection` フォルダ直下に、`.env.example` をコピーして `.env` または `.env.local` という名前で保存する
   - コマンド例（ターミナルでプロジェクトフォルダにいる場合）:
     ```bash
     cp .env.example .env
     ```
   - エディタで `.env` を開き、次のように**実際の値**に書き換える（引用符は不要でよい）:
     ```
     VITE_SUPABASE_URL=https://あなたのプロジェクトID.supabase.co
     VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.実際の長い文字列...
     ```

3. **開発サーバーを再起動**
   - 環境変数は起動時に読み込まれるため、`.env` を追加・変更したあとは `npm run dev` を一度止めて（Ctrl+C）再度 `npm run dev` を実行する

4. **動作確認**
   - ブラウザでアプリを開き、「Supabase 接続設定済み」と表示されれば Step 8 は完了です。未設定のときは「Supabase は未設定です（.env を参照）」と表示されます。

**ここまでで**: フロントから Supabase に接続する準備ができています。Step 9 でルーム作成画面から実際に DB を触ります。

## Step 9: ルーム作成画面を作る

TOP 画面から「ルームを作る」を選び、ルーム名・パスコード・メインファシリテーターのタイプ（16タイプ）・イベント日付・名前を入力して Supabase の **rooms** に保存します。作成後、参加用URLとパスコードを表示して共有できるようにしています。

### 用語（このステップで出てくるもの）

| 用語 | 説明 |
|------|------|
| **ルーム** | 1回の振り返り単位。ルームごとに URL とパスコードで参加者を集め、評価を集計します。 |
| **passcode_hash** | パスコードを bcrypt でハッシュ化した値。平文は保存せず、照合時のみ使います。 |
| **main_facilitator_type_id** | 診断アプリ「ファシリテータースタイル」の16タイプの id（例: conductor）。 |

### 手順（実装済みの内容）

- **TOP 画面（/）**: 「ルームを作る」「参加用URLで参加」の2つの入口を表示
- **ルーム作成画面（/create）**: ルーム名・パスコード・メインファシリタイプ（必須）、イベント日付・名前（任意）を入力し、送信で rooms に INSERT。パスコードはフロントで bcrypt ハッシュ化してから保存
- **作成完了後**: 参加用URL（例: `https://.../room/{roomId}/join`）とパスコードを表示し、コピーボタンで URL をコピー可能
- **参加用URLで参加（/join）**: 現時点は説明のみ。次のステップで、URL 入力または `/room/:roomId/join` でパスコード入力・役割選択を実装予定

### 動作確認

1. `npm run dev` で起動し、ブラウザで http://localhost:5173/ を開く
2. 「ルームを作る」をクリックし、ルーム名・パスコード・タイプを入力して「ルームを作成」を押す
3. Supabase の **Table Editor** で **rooms** に1行追加され、作成完了画面で参加用URLとパスコードが表示されれば Step 9 は完了です

**ここまでで**: ルームを作成し、DB に保存して参加用URLを共有できる状態になっています。

## Step 10: 参加画面（パスコード入力 → 照合 → 役割選択）

参加用URL（`/room/:roomId/join`）で開いたときに、パスコードを入力し、照合できたら役割（メイン／サブ／参加者）を選んで評価入力画面へ進む流れを実装しました。

### 用語（このステップで出てくるもの）

| 用語 | 説明 |
|------|------|
| **照合** | 入力されたパスコードと、DB に保存したハッシュ（passcode_hash）を bcrypt で比較し、一致するか確認すること。 |
| **役割** | main（メインで行ったファシリ／自己評価）、sub（サブファシリ）、participant（参加者）のいずれか。評価送信時に evaluations.role に保存する。 |

### 手順（実装済みの内容）

- **パスコード入力**: `/room/:roomId/join` でパスコード入力欄を表示。送信時に rooms から当該 room_id の行を取得（id, name, passcode_hash）し、bcrypt.compare で照合。
- **照合失敗**: 「ルームが見つかりません」または「パスコードが違います」を表示。
- **照合成功**: 役割選択画面を表示（メイン／サブ／参加者の3つ）。選択すると roomId・roomName・role を sessionStorage に保存し、`/room/:roomId/evaluate` へ遷移。
- **評価入力画面**: 現時点はプレースホルダー（「評価入力は次のステップで実装します」）。Step 11（Phase 3）で18項目のスコア＋自由記述を実装予定。

### RLS の確認（参加時に「ルームが見つかりません」になる場合）

参加時、アプリは **anon** で `rooms` の `id, name, passcode_hash` を **SELECT** して照合しています。RLS で anon が rooms を SELECT できない設定になっていると、「ルームが見つかりません」と表示されます。その場合は Supabase の **Authentication > Policies** で、**rooms** に対し「anon が 1 行を SELECT 可能」なポリシーを追加してください（例: `id = 指定した room_id` に限定するポリシー。本番では RPC で照合する方式に変更することを推奨）。

### 動作確認

1. Step 9 で作成した参加用URL（例: http://localhost:5173/room/xxxx/join）を別タブまたはシークレットで開く。
2. ルーム作成時に設定したパスコードを入力し「参加する」を押す。
3. 役割選択が表示され、いずれかを選ぶと「評価を入力する」画面（プレースホルダー）に遷移すれば Step 10 は完了です。

**ここまでで**: 参加用URL ＋ パスコードでルームに参加し、役割を選んで評価入力へ進む流れがつながっています。

## Step 11: 評価入力画面（18項目のスコア＋自由記述）

参加・役割選択後に表示される評価入力画面で、05_evaluation_items.md に定義した **6軸にグルーピングした18項目（E1〜E4, B1〜B10, F1〜F4）** を **5段階** で入力し、任意で表示名・自由記述を付けて **evaluations** に保存します。

### 用語（このステップで出てくるもの）

| 用語 | 説明 |
|------|------|
| **scores** | 項目IDをキーにスコア（1〜5）を格納する JSON。例: `{"E1": 4, "B1": 5, "F4": 3, ...}` |
| **evaluator_label** | 任意。匿名の場合は空。ニックネームなどを入れると結果表示時に「誰の評価か」の参考になる。 |
| **free_comment** | 任意。自由記述欄の内容。 |

### 手順（実装済みの内容）

- **評価項目データ**: `src/data/evaluationItems.ts` に6軸ごとの項目（id・文言）を定義。
- **入力フォーム**: 表示名（任意）、6軸でグループ化した18項目の各 1〜5 ラジオ、自由記述（任意）、送信ボタン。
- **送信**: 全項目が 1〜5 で選択されていることを確認し、`room_id`, `role`, `evaluator_label`, `scores`, `free_comment` を **evaluations** に INSERT。送信後は「送信しました」と表示し、「結果を見る」で `/room/:roomId/result`（Step 12 で実装予定）へ。
- **結果画面**: `/room/:roomId/result` は現時点でプレースホルダー（「集計・振り返りの問いは次のステップで実装」）。

### 動作確認

1. 参加用URLで参加し、役割を選んで評価入力画面へ。
2. 全18項目に 1〜5 を選択し、任意で表示名・自由記述を入力して「送信する」。
3. 「送信しました」が表示され、Supabase の **Table Editor** の **evaluations** に 1 行追加されていれば Step 11 は完了です。

**「送信しました」が出ない場合**  
送信すると画面上部に赤いエラーが表示されます。多くは **evaluations テーブルで anon が INSERT できない**（RLS ポリシー未設定）が原因です。Supabase ダッシュボードで **Authentication > Policies** を開き、**evaluations** テーブルに「anon が INSERT 可能」なポリシーを追加してください。例: ポリシー名 `anon_insert_evaluations`、操作 **INSERT**、ターゲット **anon**、WITH CHECK 式を `true`（とりあえず全行許可）で作成。本番では「パスコード検証済みのルームに限定」などに絞ることを推奨します。

**ここまでで**: 評価入力から DB 保存まで一通り動く状態です。次は結果表示（集計・振り返りの問い）の実装です。

---

## Step 12: 結果表示（集計・振り返りの問い・アクション提案）

`/room/:roomId/result` で、当該ルームの **evaluations** を取得し、**6軸の集計**と**レーダーチャート**、**reportAgent レポート**（3セクション：メインファシリ評価・6観点の点数解説・振り返りの問い・シェアドリーダーシップ）を表示します。詳細は 14_report_structure.md。

### 用語（このステップで出てくるもの）

| 用語 | 説明 |
|------|------|
| **集計** | 同一 room_id の evaluations の scores を項目（E1〜F4）ごとに平均した値。 |
| **振り返りの問い** | メインファシリのタイプに合わせた、振り返り用の問い（固定文言）。 |
| **アクション提案** | タイプに合わせた「次に試すとよいこと」の一文（固定文言）。 |

### 手順（実装済みの内容）

- **データ**: `src/data/reflectionByType.ts` に16タイプ分の「振り返りの問い」「アクション提案」を1文ずつ定義。
- **結果画面**: roomId で **rooms**（name, main_facilitator_type_id）と **evaluations**（scores 等）を取得。評価が0件のときは「まだ評価が送信されていません」と表示。
- **集計表示**: 6軸ごとの総合点とレーダーチャート、reportAgent のレポート（総評・強み・伸びしろ・6観点解説・振り返りの問い等）を表示。
- **振り返り**: main_facilitator_type_id に対応するタイプの問い・アクションを **details/summary** で折りたたみ表示。

### RLS の確認（結果が表示されない場合）

結果画面では anon が **rooms** と **evaluations** を **SELECT** します。RLS で anon の SELECT が禁止されていると、ルーム名や集計が表示されません。Supabase で **rooms** と **evaluations** それぞれに「anon が SELECT 可能」なポリシーを追加してください（本番では「パスコード検証済み」などに限定することを推奨）。

### 動作確認

1. 評価を1件以上送信したルームの「結果を見る」、または直接 `/room/{roomId}/result` を開く。
2. ルーム名・メインファシリタイプ・6軸レーダー・レポート（3セクション）が表示される。
3. 「振り返りの問い」「アクション提案」を開くと、ルームのタイプに応じた文言が表示されれば Step 12 は完了です。

**ここまでで**: ルーム作成 → 参加 → 評価入力 → 結果表示まで、MVP の一通りの流れが動作する状態です。Phase 5（仕上げ・ブランド・環境変数整備など）は必要に応じて進めてください。

## Step 13: 結果画面のアクセス制御と入力状況の表示

結果画面（`/room/:roomId/result`）を「パスコードを知っている人だけ」が閲覧できるようにし、あわせて**送信済み件数**（メイン・サブ・参加者）を表示します。

### 手順（実装済みの内容）

- **パスコード認証**: 結果URLを開いたとき、まず「結果を表示するにはパスコードを入力してください」と入力欄を表示。正しいパスコードを入力すると、同一セッション中は再入力せずに結果を表示（sessionStorage に検証済み roomId を保持）。
- **送信済み件数**: 結果画面の上部に「送信済み: メイン ○件、サブ ○件、参加者 ○件」を表示。evaluations の role ごとの件数から算出。

### 動作確認

1. 結果URL（`/room/{roomId}/result`）を**新しいタブやシークレット**で開く（まだ検証していない状態で）。
2. パスコード入力画面が表示され、正しいパスコードで「結果を表示」を押すと集計が表示される。
3. 同じタブで再度結果URLを開いても、パスコードなしで結果が表示される（タブを閉じると再度パスコードが必要）。
4. 結果画面に「送信済み: メイン 1件、サブ 0件、参加者 0件」のような表示が出ていれば Step 13 は完了です。

**ここまでで**: 結果はパスコードで保護され、誰が何件送信したかも一目で分かる状態です。

## Step 14: 入力待ちの表示（期待人数を設定した場合）

ルーム作成時に「サブの人数」「参加者の人数」を任意で入力できるようにし、結果画面で**送信済み件数と期待人数（X/Y）**および**「あと○人の入力待ち」**を表示します。

### 前提（Supabase の rooms テーブル）

Step 14 の入力待ち表示を使う場合は、**rooms** に次の2列を追加してください（任意。追加しない場合はサブ・参加者人数は入力せず、従来どおり利用できます）。

- **expected_sub_count** … INTEGER、NULL 許可
- **expected_participant_count** … INTEGER、NULL 許可

Supabase ダッシュボード → Table Editor → rooms → 新しい列を追加、で設定できます。

### 手順（実装済みの内容）

- **ルーム作成画面**: 「サブの人数（任意）」「参加者の人数（任意）」の入力欄を追加。数値を入れるとルーム保存時に上記2列に保存（列がない場合はルーム作成でエラーになるため、使う場合は先に列を追加）。
- **結果画面**: ルームに expected_sub_count / expected_participant_count がある場合、「送信済み: メイン 1/1、サブ 0/2、参加者 0/1」のように **X/Y** 形式で表示。まだ足りない場合「あと サブ2人、参加者1人 の入力待ち」を表示。

### 動作確認

1. Supabase の **rooms** に **expected_sub_count** と **expected_participant_count** を追加（整数・NULL 可）。
2. ルーム作成で「サブの人数」に 2、「参加者の人数」に 1 を入力して作成。
3. 評価をメイン 1 件だけ送信し、結果画面を開く。
4. 「送信済み: メイン 1/1、サブ 0/2、参加者 0/1」と「あと サブ2人、参加者1人 の入力待ち」が表示されれば Step 14 は完了です。

**ここまでで**: 期待人数を設定したルームでは、結果画面で入力待ちの状況が分かります。

## Step 15: Phase 5 仕上げ（ブランド統一・重複送信防止）

全画面でブランド表記を統一し、同一役割の二重送信を防ぐ仕組みを入れました。

### 15-1. ブランド表示の統一

- **共通コンポーネント**
  - `src/components/AppFooter.tsx` … 「powered by awareness=design」を表示。任意で `className`（例: `text-center`）を渡せます。
  - `src/components/AppBrandHeading.tsx` … 「ファシリテーターリフレクション」の見出し用テキスト。
- **反映先**
  - TOP / ルーム作成 / 参加入口 / 参加（パスコード・役割選択）/ 評価入力 / 結果 の全画面で、上記コンポーネントを利用し「ファシリテーターリフレクション」と「powered by awareness=design」の有無・表記を統一しました。

### 15-2. 同一役割の重複送信防止

- **同じブラウザでの二重送信**
  - 送信成功時に `sessionStorage` に「このルーム・この役割で送信済み」を記録します。
  - 評価入力画面を再度開いたとき、そのキーがあれば「この役割ではすでに送信済みです。」と表示し、フォームは表示しません。結果画面へのリンクを表示します。
- **メインは1件まで（ルーム単位）**
  - 役割が「メイン」のとき、そのルームの evaluations で role=main が既に1件以上あるか Supabase で取得して判定します。
  - 既に1件以上あれば「メインの自己評価はすでに送信済みです。」と表示し、フォームは表示しません（別デバイス・別タブでメインが既に送信した場合も防止）。
- **サブ・参加者**
  - 同じブラウザで同じ役割で再送信しようとした場合のみ、上記 sessionStorage で送信済みとみなして入力を抑制します。サブ・参加者は複数人いる想定のため、DB上では「1ルーム1件まで」などの制限はしていません。

### 動作確認

1. **ブランド**: 各画面を開き、「ファシリテーターリフレクション」と「powered by awareness=design」が表示されていることを確認。
2. **メイン二重防止**: ルームを作成し、メインで評価を1回送信。同じルームの評価入力URLを再度開き、メインで参加すると「メインの自己評価はすでに送信済みです。」と表示され、フォームが出ないこと。
3. **同一ブラウザでの送信済み**: サブで評価送信後、同じブラウザで同じルームの評価入力に戻り、再度サブで参加すると「この役割ではすでに送信済みです。」と表示され、フォームが出ないこと。

**ここまでで**: Phase 5 の仕上げとして、ブランド統一とメイン1件・同一ブラウザ同一役割の重複送信防止が入っています。

## Step 16: 軽微なUI確認・参加者上限（任意）

### 16-1〜16-4: UI確認（実施済み）

### 16-5: サブ・参加者上限

- **ルーム作成時に「サブの人数」「参加者の人数」を入力した場合**、それぞれを**その役割の評価入力の上限**として使います。
