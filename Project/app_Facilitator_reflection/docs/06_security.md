# ファシリテーターリフレクション 非機能要件・セキュリティ

## 1. 方針

- アプリのURLは一般公開するが、**ルーム単位でデータを分離**し、**鍵（パスコード）を知る者のみ**が当該ルームの参加・結果閲覧ができるようにする
- 無料枠（Supabase 等）の範囲で、**パスコードベースのアクセス制御**を採用する。ユーザー個別のログイン（メール認証等）は MVP では必須としない
- セキュリティ要件を満たしつつ、利用者が「ルームを作ってURLとパスコードを共有する」だけで使える簡易さを保つ

## 2. 認証・アクセス制御

- **ルームへのアクセス**: 参加用URL（ルームID 含む）＋ **パスコード** の入力で、当該ルームへの参加および評価入力・結果閲覧を許可する
- **同一ルーム内**: メインファシリ・サブファシリ・参加者の区別は **役割選択** で行い、ルーム内の全員が同じ結果を閲覧できる（要件 2.4 のとおり）
- **他ルームのデータ**: 参照・更新不可。データベースでは RLS（Row Level Security）により、パスコード検証済みのセッションが持つ room_id に紐づく行のみ読み書き可能とする（詳細は [04_data_design.md](04_data_design.md) の RLS 方針を参照）

## 3. パスコードの扱い

- **保存**: パスコードは **平文で保存しない**。bcrypt 等でハッシュ化した値を `rooms.passcode_hash` に格納する
- **照合**: 参加時に入力されたパスコードをハッシュと照合し、一致した場合にのみ当該ルームへのアクセスを許可する
- **クライアント**: パスコードを長時間保持しない。検証後にセッション用トークンやルームアクセス許可状態のみを保持する形とする（実装方針）

## 4. 通信

- **HTTPS**: フロントエンド・API・データベースへの通信はすべて **HTTPS** とする。Vercel・Supabase 等の利用で標準的に満たす想定

## 5. 秘密情報の取り扱い

- **API キー**: Supabase の `service_role` キーは **サーバー側のみ** で使用し、フロントエンドやリポジトリに含めない。フロントエンドで使用するのは `anon` キーおよび RLS による制限付きアクセスのみ
- **環境変数**: 接続情報・API キーは環境変数で管理し、リポジトリにコミットしない

## 6. データ最小化・個人情報

- **評価者の特定**: 評価入力時の表示名（ニックネーム）は任意とし、匿名での回答も可能とする。個人を特定する情報は必要最小限に留める
- **ルームの有効期限**: 実装時に、ルーム作成から一定期間経過後の扱い（閲覧のみとするか、削除するか）を検討する
