# レポートエージェント コメント生成仕様書（11_report_agent）

ワークショップ振り返りコメントを**高品質に安定生成する**ための、コメント生成専用の仕様を定義する。実装コードは本ドキュメント確定後に別フェーズで行う。

**前提**: [01_requirements.md](01_requirements.md) 2.4.1 に準拠する。分析結果文章・振り返り用の問い・アクション提案は**箇条書き**で表示する。本アプリの目的は**問いを通じた関係性の深化と行動変容の支援**であり、満足度評価ではなく**感情の動き**と**実践可能性**に焦点を当てる。振り返りの問いの拡張（シェアドリーダーシップ・知覚スキーマ認知）は [03_requirements_reflection_shared_leadership_cognition.md](03_requirements_reflection_shared_leadership_cognition.md) で定義しており、実装時には `reflectionQuestions` の件数拡張または新キー（例: `sharedLeadershipReflectionQuestions`）の追加は 03 の要件に従う。

---

## 1. 目的と前提

### 1.1 目的

- 振り返りコメントの**品質と一貫性**を担保するため、生成時の入力・ルール・出力形式を本仕様で確定する
- **テンプレ方式のみで合格品質を満たす**ことを最優先とする。AI は言い回しの自然化（polish）など上振れ用途とし、必須依存にしない
- レポート生成時は**生データではなく分析済みデータ**を渡し、本仕様に従ってコメントを生成する
- 出力は**満足度の要約ではなく、感情の動きの読み解きと実践可能な次の一手**に寄せる

### 1.2 前提（01 2.4.1 準拠）

- **表示構成**: **六角形レーダー**（6軸）→ **6軸**の総合点数と分析結果（箇条書き）→ 振り返り用の対話の問い（箇条書き）→ アクションの提案（説明＋箇条書き3点程度）
- **分析結果文章**: **各軸（6軸）**について、総合点数と分析コメントを**箇条書き**で表示する
- **振り返り用の問い**: 3問と意図を**箇条書き**で表示。約1時間の振り返り想定。拡張時（03 シェアドリーダーシップ・知覚スキーマ認知）は、問いの追加または別ブロックの追加で 03 の要件に従う
- **アクションの提案**: 内容の説明文章と、**箇条書きでアクションを3点程度**表示する

## 3. 入力データ仕様（AI に渡す「分析済みデータ」）

AI には**生データ（評価行の scores の羅列）ではなく、分析済みデータのみ**を渡す。

### 3.1 必須の分析済みデータ項目

コメント生成に渡す分析済みデータには、次の項目を**必ず**含める。

| 必須項目 | 説明 |
|----------|------|
| **n（回答者数）** | 評価が有効な回答者数。役割内訳（メイン/サブ/参加者ごとの人数）を含めてもよい。 |
| **全体平均** | 全評価・全項目を合わせた平均（1〜5 のスコアの全体平均）。 |
| **6軸の総合点（平均）＋平均との差** | 各軸（説明・設計、見える化・編集、場の観察、場のホールド・安心感、問いかけ・リフレーミング、流れ・即興）のセクション平均。さらに各軸の平均と**全体平均との差**を付与する。 |
| **最高の軸と最低の軸（平均との差）** | 6軸のうち、セクション平均が**最も高い軸**と**最も低い軸**を特定し、それぞれの平均との差を渡す。 |
| **ばらつきが大きい軸（あれば）** | 標準偏差または IQR が大きい軸をリストする。該当がなければ空配列または省略可。 |
| **役割別の差分が大きい軸（あれば）** | メイン・サブ・参加者の役割別セクション平均の差が大きい軸をリストする。該当がなければ空配列または省略可。 |
| **状態タイプ分類** | セクション平均や役割間差のパターンから付与した類型。例：安全性高・挑戦不足型、メイン厳しめ型 など。1つ以上。 |

上記のうち「あれば」と書いた項目は、該当する場合のみ値を渡せばよい。渡す場合はキー名をスキーマに合わせて統一する。

### 3.2 JSON スキーマ例

上記必須項目を満たす形の例。キー名は実装時に揃えればよい。

- **n**: 必須。回答者数。`meta.n` または `meta.evaluationCount` のいずれかで渡す。

## 4. スコア帯 → 意味ラベル

セクション平均（または役割別平均）を次の帯にマッピングし、意味ラベルを付与する。コメント生成時に「どの帯か」を参照する。

| スコア帯（セクション平均） | band ID | 意味ラベル |
|----------------------------|---------|------------|
| 4.5 以上                   | very_high | 非常に高い |
| 4.0 以上 4.5 未満          | high      | 高い       |
| 3.0 以上 4.0 未満          | standard  | 標準       |
| 2.5 以上 3.0 未満          | improvement | 改善余地   |
| 2.5 未満                   | attention  | 要対応     |

- 役割間の**相対差**（メインが他より 0.5 以上高い/低い、役割間で近い）は、分析済みデータの stateTypes や役割別平均で表現する
- 意味ラベルは「満足度」ではなく、**感情の動き・実践の手がかり**として使う（例: 「非常に高い」→ 安心感や届き方が強く出ている可能性、「改善余地」→ 挑戦や編集に余裕がなかった可能性）

## 5. コメント生成ルール

### 5.1 必須

- **相対差を必ず言及する**: 平均との差、最高・最低項目、役割間の差（メイン vs サブ vs 参加者）のいずれかを必ず触れ、数値の羅列だけで終わらせない
- **強み・改善仮説・次回アクションを必ず含める**: 出力フォーマットの summary / strengths / improvementHypotheses / nextActions をすべて埋める。強みだけ・改善だけに偏らない
- **抽象語だけで終わらせない**: 「良かった」「学びがあった」「よかったです」等は禁止。具体的な観察・推論・アクションで書く

### 5.2 推奨

- **感情の動きの読み解き**: 点差から「安心感がどうだったか」「挑戦の感じ方」「届き方の認識差」を仮説として述べる
- **背景仮説を立てる**: 「こういう感情や状況が起きていたのではないか」と明示する。断定せず仮説として表現する
- **実践可能なアクション**: 誰が・いつ・何をするかが分かる形で2点（以上）提示する

## 6. 6軸ごとの示唆テンプレ辞書（各スコア帯 × バリエーション）

6軸（説明・設計、見える化・編集、場の観察、場のホールド・安心感、問いかけ・リフレーミング、流れ・即興）それぞれについて、スコア帯（band）と状態タイプに応じた文言を定義する。実装は [05_evaluation_items.md](05_evaluation_items.md) の6軸名称に合わせ、[sectionCommentTemplates](../src/lib/reportAgent/sectionCommentTemplates.ts) 等で band × 軸ごとに複数バリエーションを配列で持つ。同じ band でも複数文を用意し、§8 の決定的選択（seed 固定）でどれを出すか決める。ランダム選択は行わない。

## 7. テンプレ辞書のデータ構造

band（§4）× section（strengths / improvementHypotheses / nextActions / sectionComments）× **軸**（sectionComments のみ。design, visibility, observation, hold, questioning, flow）単位で配列を定義する。

- **sectionComments**: 6軸キー（design, visibility, observation, hold, questioning, flow）× band → 文字列配列。各軸・各 band ごとに複数文を配列で持ち、選択時に 2 本選ぶ。
- **strengths / improvementHypotheses**: 状態タイプ × ブロック → 文字列配列。状態タイプごとに最大3文まで。
- **nextActions**: 状態タイプ × ブロック → `{ summary: string, bullets: string[] }`。bullets は2本固定（UI は3まで許容）。

以下はスキーマ例（sectionComments は6軸キー。実装は sectionCommentTemplates.ts を参照）。

## 8. テンプレ選択仕様（決定的選択）

テンプレからの選び方は**ランダム禁止**とする。**同じ入力なら常に同じ出力**になることを要件とする。

- **seed**: 次のいずれかを連結した文字列で固定する。
  - **sectionComments**: `roomId + section("sectionComments") + band + stateType + axis`（axis は design, visibility, observation, hold, questioning, flow）
  - **strengths / improvementHypotheses / nextActions**: `roomId + section + band または stateType`（実装で section 名と stateType を組み合わせる）
- seed から配列の**インデックス**を決定的に算出し、その位置の要素を選ぶ。必要な個数（例: sectionComments は各視点 2 本）だけ、seed に連番を足すなどして重複しないインデックスを生成する。

疑似コード例:

```
function selectIndex(arr: string[], seed: string): number
  hash = 0
  for each char c in seed: hash = (hash * 31 + c) mod 2^32
  return abs(hash) mod arr.length

function pickBullets(arr: string[], n: number, seed: string): string[]
  result = []
  for i in 0..n-1:
    idx = selectIndex(arr, seed + str(i))
    result.push(arr[idx])
  return result
```

- 実装時は `roomId`・`band`・`stateType`・`viewpoint` をすべて同じルールで文字列連結し、上記のように hash から index を求める。これにより、同じルーム・同じ分析結果であれば必ず同じコメントが生成される。

## 9. 重複回避・文字数制約（UI 表示を想定）

出力の重複を避け、表示量をそろえるため、次のルールを満たす。

| ブロック | ルール |
|----------|--------|
| **sectionComments** | 各軸（design, visibility, observation, hold, questioning, flow）の bullets は**2本固定**。他ブロックと同一文を繰り返さない。 |
| **strengths** | 最大**3 bullets**。summary や sectionComments と重複する文言にしない。 |
| **improvementHypotheses** | 最大**3 bullets**。同上。 |
| **nextActions** | **2 bullets 固定**（多くても 3 まで。UI は 3 まで許容）。 |
| **summary** | **全体の結論＋相対差の一言**に限定する。factSentence で事実（数値・役割・最高/最低視点）を1文で述べ、bullets は他ブロックと同じ内容を繰り返さない。 |

- 文字数上限は別仕様（品質ゲート・truncate）に従う。本節は「何を何本出すか」の構造制約とする。

## 10. AI 自然化（polish）の安全要件

AI で言い回しを整える場合でも、以下を満たす。**テンプレ方式のみで合格品質を満たす**ことが最優先であり、AI は品質の上振れ用途とする。AI に必須依存しない。

| 要件 | 内容 |
|------|------|
| **意味・数値を変えない** | スコア・役割名・視点名・高低・差の事実は増減・変更禁止。言い換えても意味が同一であること。 |
| **bullets の数を変えない** | 各 `bullets` 配列の要素数は入力と同一とする。結合・分割・追加・削除禁止。 |
| **推測は推測のまま** | 「〜の可能性」「〜と読める」等を断定形に書き換えない。 |
| **出力 JSON キー構造は固定** | §2 のキー（summary, strengths, improvementHypotheses, nextActions, sectionComments, reflectionQuestions, actionProposal）と同一構造で返す。 |
| **polish 後の検証とフォールバック** | polish 適用後、品質ゲート（validate）を通す。不合格の場合は **テンプレ出力にフォールバック**し、テンプレをそのまま返す。 |

- 詳細は [13_quality_gate_and_polish.md](13_quality_gate_and_polish.md) の「AI 言い回し整形」を参照する。

## 11. 振り返りの問い（3問＋意図、箇条書き、約1時間想定）

以下を箇条書きで表示する。編集者が文言を更新可能とする。

1. **問い**: 今回の場で、メインとして「うまくいった」と感じた瞬間はどこでしたか？ 参加者・サブの皆さんは、そのとき何を感じていましたか？  
   **意図**: 自己認識と他者からの見え方のすり合わせ。次の改善の土台を共有する。

2. **問い**: 6つの観点のうち、今日の振り返りで特に話したい観点はどれですか？ その観点で、メインに伝えたいことがあれば教えてください。  
   **意図**: 6観点から優先して深掘りするテーマを選び、具体的なフィードバックを引き出す。

3. **問い**: 次のファシリテーションで、メインが「1つだけ試してみる」としたら、何を試すと良さそうですか？ 参加者・サブの視点も含めて一言ずつ。  
   **意図**: 具体的な次のアクションを対話で決める。約1時間の振り返りの締めとして実行可能な一歩に落とす。

## 12. アクション提案（説明＋箇条書き3点程度）

- **説明**: なぜこのアクションが提案されるか、分析結果（強み・改善仮説）とどう結びつくかを短く述べる。満足度の要約ではなく、感情の動きや相対差から得た示唆に基づく。
- **箇条書き**: 実行可能なアクションを**3点程度**、誰が・いつ・何をするかが分かる形で書く。抽象的な「意識する」だけで終わらせず、具体的な「1回試す」「一言聞く」等を含める。

例（説明）: 「流れの見え方の差と、安心感の届き方を次回に活かすため、時間の区切りと一言確認を1つずつ試す。」

例（箇条書き）:
- 時間の区切りを入れるタイミングを1つ決め、終了後に参加者に「流れはどう感じたか」を一言聞く。
- 「安心して話せた瞬間」を参加者に1つ挙げてもらい、メインは聴くだけの時間を取る。
- 振り返りで「余白を取れた／取れなかった」を参加者と一言共有する。

## 13. 禁止事項・トーン

### 9.1 禁止

- **研修っぽい一般論**: 「ファシリテーションでは〇〇が重要です」「日頃から△△を心がけましょう」など、本ルームのデータと無関係な一般論で締めない
- **根拠のない断定**: 分析済みデータにないことを断定しない。推測は「〜の可能性」「〜と読める」等で**推測であると明示**する
- **満足度評価的な要約**: 「よかった」「満足でした」や、スコアの良し悪しの羅列のみで終わらせない
- **抽象語のみ**: 「良かった」「学びがあった」等だけでは書かない。具体的な観察・仮説・アクションを添える

### 9.2 トーン

- **組織開発・関係性デザインの専門コンサルタント**として、関係性とプロセスの観点で読み解く
- **問いを通じた関係性の深化と行動変容**に寄せる。感情の動きと実践可能性を前面に出す
- 断定より**仮説**。「〜の可能性」「〜と解釈できる」を適宜使う
- **メインファシリテーターのモチベーション**: レポートの主な読み手はメインファシリ。強みは**肯定を前面に**（「高く評価されています」「届いていたと読めます」など）、伸びしろは**未来志向**（「伸ばしやすい領域」「次に手を入れやすい」）で表現する。批判・課題は**事実＋解釈**で残し、断定は避ける。
- **要点の見せ方**: 総評は短い factSentence ＋ bullets 2本で「傾向」「次の一手」を分ける。強み・伸びしろは1本目を要約文にし、2本目以降で補足する。観点別コメントは1文目で結論、2文目で理由とする。

## 14. 入力例 → 出力例（2ケース）

### ケース1: 安全性高・挑戦不足型

**入力例（分析済みデータ要約）**

- 役割: メイン1、参加者2。サブなし
- 6軸のセクション平均: 最高軸 hold（場のホールド・安心感）4.2、最低軸 flow（流れ・即興）3.5
- 役割別: メインは hold 高め・flow 低め、参加者も同傾向
- 状態タイプ: safety_high_challenge_low（安全性高・挑戦不足型）
- スコア帯: hold=very_high、flow=improvement

**出力例（要約）**

- **総評**: 場のホールド・安心感が最も高く評価され、流れ・即興は相対的に低めで伸ばせる領域と読めます。メインと参加者で傾向が近い。
- **強み**: 場のホールド・安心感が役割を超えて共有されていると読める。参加者観察・安心感に手が入っている。
- **改善仮説**: 流れ・即興が improvement 帯。流れの編集や即興的な軌道修正に余裕がなかった可能性。
- **次回アクション**: ① 時間の区切りを1つ決め、終了後に参加者に「流れはどう感じたか」を一言聞く。② 状況に合わせて進め方や問いを変えるタイミングを1つだけ意図して試す。

### ケース2: 説明・設計が低め・場の観察が高め

**入力例（分析済みデータ要約）**

- 役割: メイン1、サブ1、参加者2
- 6軸のセクション平均: 最低軸 design（説明・設計）3.5、最高軸 observation / hold 付近 4.0
- 状態タイプ: 全体設計・伝達は弱め、参加者観察は強め
- スコア帯: design=improvement、observation=high、flow=standard

**出力例（要約）**

- **総評**: 説明・設計が相対的に低く、場の観察・安心感は役割間で4前後と高い。目的・進め方・問いの伝達に余裕がなかった一方、参加者の反応を捉えて場に活かしていたと複数役割が感じている。
- **強み**: 場の観察・問いかけが高く、安心感や問いの届き方が共有されている可能性。
- **改善仮説**: 説明・設計が improvement 帯。設計や説明の伝達が参加者に届き切っていない可能性。
- **次回アクション**: ① 目的・進め方・問いを「1文で言うと」と参加者に確認する瞬間を1つ設ける。② 振り返りで「どこが伝わったか・伝わらなかったか」をサブ・参加者に一言ずつ聞く。
