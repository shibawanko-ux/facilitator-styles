# 06 GAS連携（個人用スライド生成）

**対象は Phase 2 および Phase 3 のみとする。** Phase 1（実施前のみ）では GAS の出力を行わず、Phase 1 だけに関するアウトプットは必須としない。

Phase 2/Phase 3 の個人別スライドを自動生成する GAS コードと、本アプリが出力するデータの対応を定義する。

## 1. 目的

- 各参加者ごとに1枚のスライド（`radar_chart_P`）を生成する。
- 表紙（`cover`）は1回のみ生成する。

## 2. プレースホルダー形式

| 種別 | 形式 | 例 |
|------|------|-----|
| 表紙用 | `{{XXX}}` | `{{Client}}`, `{{date}}` |
| 個人用 | `{{XXX_P}}` | `{{name_P}}`, `{{CgrA_1_P}}`, `{{CgrF_1_P}}`（総合スコア。17 参照） |
| Phase3 上長・ギャップ | `{{Ggr{letter}_2_P}}`, `{{Ggr{letter}_3_P}}` | 上長評価・ギャップ値 |

## 3. レイアウト

- **LAYOUT ID**: `cover`, `radar_chart_P`
- テンプレートスライドIDは環境ごとに設定する（Phase2/Phase3 で異なるIDを参照可能とする）。

## 4. スライド生成の2段階アプローチ

1. **準備**: 元の `radar_chart_P` スライドを（参加者数−1）回複製し、表紙は既存の `cover` を1回使用。スライド配列を用意する。
2. **データ入力**: 用意したスライド配列と参加者データを順番に対応づけ、各スライドに対して `replacePlaceholdersInSlide_` でプレースホルダーを置換する。
- **重要**: 先にすべてのスライドを複製してから置換する。置換済みスライドを複製すると、プレースホルダーが残らず2人目以降が正しく反映されない。

## 5. 参加者データ構造（例）

```javascript
const PARTICIPANTS_DATA = [
  {
    "name": "氏名",
    "placeholders": {
      "{{name_P}}": "氏名",
      "{{CgrA_1_P}}": "2.33",
      "{{C_block_3_1_body_P}}": "湧いている",   // 活用意欲（Q16Aを入力文言で表示）
      "{{C_block_3_2_body_P}}": "アクション宣言の自由記述",
      // ... 各プレースホルダーと値
    }
  },
  // ...
];
```

### 5.1 Phase 別プレースホルダーの内容

プレースホルダー `{{C_block_3_1_body_P}}` と `{{C_block_3_2_body_P}}` はPhaseによって内容が異なる。

#### Phase 2

- `{{C_block_3_1_body_P}}`: **活用意欲**。直後.csv の Q16A を**入力文言**で表示（「非常に湧いている」「湧いている」「どちらともいえない」「湧いていない」「全く湧いていない」等）。値は `_convert_q16a_to_text` で数値→ラベルに変換する。
- `{{C_block_3_2_body_P}}`: **アクション宣言**。直後.csv の Q17A（自由記述）をそのまま表示。データがない場合は「-」。

#### Phase 3

- `{{C_block_3_1_body_P}}`: **実践エピソード**。1ヶ月後.csv の Q17B（1ヶ月後の実践エピソード・障壁、自由記述）をそのまま表示。データがない場合は「-」。
- `{{C_block_3_2_body_P}}`: **上長コメント**。上長1ヶ月後.csv の M7（上長コメント、自由記述）をそのまま表示。データがない場合は「-」。

本アプリは スライド挿入内容（個別）.md を生成する際に上記を埋め、**個人用GASコード**はその Markdown を解析して `PARTICIPANTS_DATA` の各 `placeholders` に含める。テンプレート側に `{{C_block_3_1_body_P}}` / `{{C_block_3_2_body_P}}` が存在する場合、GAS の置換で反映される。

## 6. GASコードの自動生成要件

### 6.1 全体用GASコード（`GASコード_{project_name}_Phase{N}.gs`）

- **FR-GAS-AUTO-1**: CSV アップロード・分析完了時に、最新 Phase の全体用 GAS コードを**自動生成**して保存する。
- 生成タイミング: `analyze_all()` 内、分析処理完了直後。

### 6.2 個別用GASコード（`GASコード（個別）_{project_name}_Phase{N}.gs`）

- **FR-GAS-AUTO-2**: Phase 2/3 で個別スライド挿入内容（`スライド挿入内容（個別）_Phase{N}.md`）が生成された場合、**個別GASコードも同時に自動生成**して保存する。
- 生成タイミング: `analyze_all()` 内、`individual_slide_contents` 収集後にループで生成。
- 対象: `individual_slide_contents` に含まれる全 Phase（2 および 3）。
- エラー時はログ出力のみ行い、他の処理を止めない。

### 6.3 Excelファイルのサポート

- **FR-GAS-AUTO-3**: `.xlsx` / `.xls` 形式でアップロードされたファイルは、`save_uploaded_file()` で CSV（UTF-8-sig）に自動変換してから保存する。ユーザーは Excel を加工せずそのままアップロードできる。

## 8. 参照（旧 6）

- 詳細な入力Markdown形式・出力GASコード要件・テンプレートスライドIDは、本ドキュメントと 05_分析と出力仕様 を参照する。
- 本アプリは、分析結果から上記 `PARTICIPANTS_DATA` 相当のデータを生成し、GAS で利用可能な形で出力することを目指す。
- **スキル分析テーブル内の総合スコア行**（CgrF_*_P 等）: [17_スキル分析テーブル_総合スコア行追加要件.md](17_スキル分析テーブル_総合スコア行追加要件.md)

## 9. GASコード生成ダイアログの表示仕様

「GASコードを生成」ボタンで開くダイアログにおいて、Phase選択ボタンの表示は以下とする。

### 7.1 Phase1用ボタンの削除

- **FR-GAS-UI-1**: 「Phase 1用 GASコードを生成」ボタンは**表示しない**（削除する）。  
  Phase 1 では GAS 出力を行わない（本文セクション1）ため、UI上でも選択肢として提供しない。

### 7.2 プロジェクトのPhaseに応じたボタン表示

- **FR-GAS-UI-2**: 表示するボタンは、**そのプロジェクトで利用可能なPhase**（アップロード済みCSVに基づく Phase 判定結果）に応じて切り替える。
- **FR-GAS-UI-3**: **実施前と直後のCSVしか存在しない場合**（＝プロジェクトの Phase が 2 の場合）は、**Phase 2 用のコード生成ボタンのみ**表示する。
  - 表示するボタン:
    - 「Phase 2用 GASコードを生成」（全体用）
    - 「Phase 2用 個人GASコードを生成」（個人用）
  - 非表示とするボタン:
    - 「Phase 3用 GASコードを生成」
    - 「Phase 3用 個人GASコードを生成」
- **FR-GAS-UI-4**: **実施前・直後・1ヶ月後のCSVが存在する場合**（＝プロジェクトの Phase が 3 の場合）は、Phase 2 用・Phase 3 用の**いずれも**表示する（従来どおり）。

### 7.3 判定方法

- プロジェクトの Phase は、既存の「Phase の自動判定条件」（04_データ仕様とフェーズ）に従う。バックエンドは **reports フォルダ内の生成レポート（Phase 1/2/3 の .md）の有無**で max_phase を算出する。
- フロントエンドは、`/api/project/<project_id>/files-status` のレスポンスに含まれる **files_status.max_phase** を参照し、上記表示ルールを適用する。

### 7.4 API エラー時の表示

- **FR-GAS-UI-5**: `/api/project/<project_id>/files-status` が 4xx/5xx を返した場合、またはネットワークエラーで取得に失敗した場合は、「Phase 2 以降で利用可能です」とは**別のメッセージ**を表示する。
  - 表示例:「プロジェクトの情報を取得できませんでした。ページを再読み込みするか、しばらくして再度お試しください。」
  - これにより、分析未実行（max_phase が 1）と API エラーをユーザーが区別できるようにする。
