# 10 分析コメント品質要件

スライド挿入内容における分析コメントの文章量と納得度を規定する。読み手が分析結果に対して納得感を高められることを目的とする。

## 1. 対象プレースホルダーと要件一覧

| プレースホルダー | 文章量 | 用途 | 現在の目安 |
|----------------|--------|------|-----------|
| `{{S_block_1_body}}` | **約400文字** | エグゼクティブサマリー総評コメント | 約100〜150文字 |
| ~~`{{O_block_1_body}}`~~ | — | **廃止**。組織別分析のプレースホルダーは **13_スライド3_組織別分析_プレースホルダー仕様** を参照 | — |
| `{{O_block_2_body}}` | **約300文字** | 組織別分析・強み | 約120〜150文字 |
| `{{O_block_3_body}}` | **約300文字** | 組織別分析・弱み | 約120〜150文字 |

※ 文字数は改行・スペースを含む。目標は目安であり、±50文字程度の許容とする。

## 2. 納得度を高めるための指針

分析コメントが読み手に納得されるため、以下の要素を含める。

### 2.1 根拠の明示

- **数値の提示**: スコア、変化量、割合などを具体的に示す。
- **比較の提示**: 実施前との比較、他軸との比較、組織平均との比較など。

### 2.2 解釈の説明

- **意味の説明**: その数値や傾向が何を意味するかを簡潔に述べる。
- **文脈の付与**: 組織タイプ、フェーズ、スキルバランスなどによる位置づけ。

### 2.3 示唆・次のアクション

- **示唆**: 強みをどう活かすか、弱みをどう補うかの方向性。
- **次のアクション**: 必要に応じて、具体的な改善施策やフォローアップの提案（簡潔に）。

### 2.4 文章の構成

- **論理の流れ**: 現状 → 根拠 → 解釈 → 示唆 の順で構成することを推奨。
- **重複の回避**: 同じ表現の繰り返しを避け、情報密度を高める。

## 3. プレースホルダー別詳細要件

### 3.1 `{{S_block_1_body}}`（エグゼクティブサマリー総評）

- **文章量**: 約400文字
- **内容要素**:
  1. 全体のスコア変化（実施前→直後、必要に応じて1ヶ月後）の要約
  2. 向上幅が大きいスキル軸の具体的な数値と意味
  3. 組織のスキル特性（バランス型・実行部隊型等）の変化
  4. 学習効果の確認と、今後の期待・示唆
- **出力元**: `generate_summary_comment`（`report_generator_impl.py`）
- **形式**: 箇条書きまたは段落。スライド表示に適した改行・句読点を使用。

### 3.2 ~~`{{O_block_1_body}}`（組織別・部署の特徴）~~ 【廃止】

- 本プレースホルダーは **13_スライド3_組織別分析_プレースホルダー仕様** の適用により**廃止**した。組織別スライドでは `{{O_respondents_1}}`・`{{OgrA_1}}`〜`{{OgrE_5}}`・`{{OrkA_1}}` 等に置き換えられている。旧要件の要約は 13_スライド3_組織別分析_プレースホルダー仕様 の「廃止したプレースホルダー」節を参照。

### 3.3 `{{O_block_2_body}}`（組織別・強み）

- **文章量**: 約300文字
- **内容要素**:
  1. 強みと判定したスキル軸（最大2件）のスコアと説明
  2. その強みが業務・組織にどう貢献するかの解釈
  3. 強みを維持・強化するための示唆（1〜2文）
- **出力元**: `identify_strengths` の結果を拡張し、解釈・示唆を付加するロジック
- **形式**: 段落または改行区切りの短文。既存の「スキル名（スコア点）: 説明」形式を拡張。

### 3.4 `{{O_block_3_body}}`（組織別・弱み）

- **文章量**: 約300文字
- **内容要素**:
  1. 弱みと判定したスキル軸（最大2件）のスコアと説明
  2. その弱みが業務・組織にどう影響するかの解釈
  3. 改善の方向性や優先度の示唆（1〜2文）
- **出力元**: `identify_weaknesses` の結果を拡張し、解釈・示唆を付加するロジック
- **形式**: 段落または改行区切りの短文。既存の「スキル名（スコア点）: 説明」形式を拡張。

## 4. 実装上の注意

- **文字数制限**: スライドのレイアウト（フォントサイズ・エリア幅）を考慮し、最大文字数を超えないようチェックすることを推奨。
- **データ依存**: 全社・組織・部署ごとにデータ量が異なるため、データが少ない場合でも最低限の納得感を担保する文言とする。
- **Phase別**: Phase 1/2/3 で利用可能なデータが異なるため、Phase に応じた内容の切り替えを行う。
- **GAS連携**: スライド挿入内容Markdown から GAS コードへ埋め込む仕組みは現状のままとし、文字数・内容の変更のみを対象とする。

## 6. 活用意欲・アクション宣言の活用（Phase 2 以上）

個別参加者の **Q16A（活用意欲）** と **Q17A（アクション宣言）** を分析コメントに反映し、定量的スコアに**質的な根拠**を補足することで納得度を高める。

### 6.1 データの概要

- **Q16A（活用意欲）**: 1〜5の数値。表示文言は「非常に湧いている」「湧いている」「どちらともいえない」「湧いていない」「全く湧いていない」等。
- **Q17A（アクション宣言）**: 自由記述。参加者が「変えてみよう」「やってみよう」と思う具体的な行動。
- **データソース**: 直後.csv。所属部署と紐づけて部署別に集計可能。

### 6.2 活用方針

| プレースホルダー | 活用内容 |
|----------------|----------|
| `{{S_block_1_body}}` | 全社の活用意欲集計（湧いている以上: ○名/○名、割合）＋代表的なアクション宣言1〜2件の要約・引用 |
| ~~O_block_1_body~~ | **廃止**（13 参照） |
| `{{O_block_2_body}}` | 強み領域に関連するアクション宣言の代表例（該当部署・参加者から抽出）を1件程度引用 |
| `{{O_block_3_body}}` | 弱み領域の改善を志向するアクション宣言の代表例を1件程度引用 |

### 6.3 具体的な要件

#### 6.3.1 `{{S_block_1_body}}` への組み込み

- **活用意欲の集計**: 参加者全員の Q16A を集計し、「湧いている」以上（数値4以上に相当）の人数と割合を算出。
- **記載例**: 「参加者20名のうち16名（80%）が活用意欲「湧いている」以上と回答し、現場での実践意欲が高い。」
- **アクション宣言の引用**: 全参加者から代表的な Q17A を1〜2件選び、要約して引用。選定基準は「具体的で業務に直結している」「スキル向上・実践と結びついている」等。
- **記載例**: 「代表的なアクション宣言として「インタビュー時に価値観の深掘りを意識する」「ファクトからペルソナをつくる工程を実務で実践する」等が見られ、学習内容の定着が期待される。」

#### 6.3.2 ~~`{{O_block_1_body}}` への組み込み~~ 【廃止】

- `{{O_block_1_body}}` は廃止のため、本節は適用しない。組織別スライドのプレースホルダーは **13_スライド3_組織別分析_プレースホルダー仕様** を参照。

#### 6.3.3 `{{O_block_2_body}}` への組み込み

- **強みと関連する宣言の選定**: 強みと判定したスキル軸（例: リサーチ・分析力、具体化・検証力）に関連するキーワードを含む Q17A を検索。
  - 例: リサーチ・分析力 → 「インタビュー」「ヒアリング」「深掘り」「分析」等
- **引用方針**: 該当する宣言が複数ある場合は、その部署の参加者を優先。1件を要約（最大80文字程度）して「参加者からは「〜」といったアクション宣言も見られ、強みの現場活用が期待される。」のように付加。

#### 6.3.4 `{{O_block_3_body}}` への組み込み

- **弱み改善を志向する宣言の選定**: 弱みと判定したスキル軸の向上を志向する Q17A を検索。
  - 例: 実現・ディレクション力が弱い部署で「チーム内のコミュニケーションを増やして目線を合わせたい」等
- **引用方針**: 弱み領域の改善に資する宣言を1件選び要約。「参加者からは「〜」といった改善を志向した宣言があり、補強の機運がうかがえる。」のように付加。

### 6.4 選定・引用時の注意

- **文字数制限**: アクション宣言の引用は、各ブロックの目標文字数（S_block_1: 約400文字、O_block_2/3: 約300文字）を超えないよう調整。引用部分は各ブロック内で50〜100文字程度を目安とする。（O_block_1 は廃止）
- **要約**: Q17A が長い場合（100文字超など）は、要点を抽出して要約する。原文の意図を損なわない範囲で簡潔に。
- **個人の特定を避ける**: 引用時は「参加者から」「ある参加者は」等とし、氏名は出さない。スライド挿入内容（全体用）は組織・部署レベルで共有されるため。
- **データ不足時**: Q16A・Q17A が存在しない、または該当する宣言がない場合は、従来のスコアベースの記述のみで構成する。欠損時もエラーにせず、利用可能な情報だけで品質を担保する。

### 6.5 実装上のデータフロー

- **入力**: `post_data`（直後.csv 正規化後）。各行に `Q16A`, `Q17A`, `所属部署`（または `﻿所属部署`）が含まれる。
- **集計**: 全社・部署ごとに Q16A の分布を集計。Q17A は部署・スキル軸との紐づけで検索・選定するロジックを追加。
- **キーワードマッピング**: スキル軸と Q17A の関連付けのため、スキル軸ごとに検索キーワードを定義（例: リサーチ・分析力 → インタビュー、ヒアリング、深掘り、分析、ペルソナ、等）。完全一致でなく部分一致で検索してよい。
- **変更対象**: `generate_summary_comment_impl` に `post_data` を渡す。O_block_2/3 の生成時に `post_data` と部署情報を参照する拡張を行う。（O_block_1 用の `generate_department_characteristics_impl` は廃止に伴い削除）

### 6.6 Phase 別の適用

- **Phase 1**: 直後.csv が存在しないため、活用意欲・アクション宣言の活用は行わない。
- **Phase 2**: 上記すべての活用を適用する。
- **Phase 3**: Phase 2 と同様に適用。加えて、1ヶ月後データ（Q16B, Q17B）があれば、定着状況とのクロス参照も検討可能（将来拡張）。

## 7. 満足度・理解度の活用（Phase 2 以上）

ワークショップに対する **WS満足度** と **WS理解度**（および **NPS**）を分析コメントに反映し、スコア向上が**学習効果**によるものであることを補強する。満足度・理解度が高いことは、ワークショップの内容が適切に伝達されたことを示し、スキルスコアの改善の信頼性を高める。

### 7.1 データの概要

- **WS満足度**: 直後.csv「今回のワークショップの内容に対する満足度はどの程度ですか？」。1〜5の数値（5=大変満足）。
- **WS理解度**: 直後.csv「今回のワークショップの内容をどの程度理解できましたか？」。1〜5の数値（5=深く理解できた）。
- **NPS**: 直後.csv「このワークショップを、同僚や知人にどの程度おすすめしたいですか？」。0〜10の数値。
- **データソース**: 直後.csv。04_満足度分析.csv に全社平均が集計される。所属部署と紐づけて部署別集計も可能。

### 7.2 活用方針

| プレースホルダー | 活用内容 |
|----------------|----------|
| `{{S_block_1_body}}` | 全社の満足度・理解度（平均値）を記載し、学習内容の伝達が適切であったこと、スコア向上の信頼性を補強 |
| ~~O_block_1_body~~ | **廃止**。組織別の理解度は `{{OrkA_1}}` で表示（13 参照） |
| `{{O_block_2_body}}` | 満足度・理解度が高い部署では、強みの定着が期待される旨を1文付加（オプション） |
| `{{O_block_3_body}}` | 満足度・理解度が高い部署では、改善意欲が高い旨を1文付加（オプション） |

### 7.3 具体的な要件

#### 7.3.1 `{{S_block_1_body}}` への組み込み

- **全社の満足度・理解度**: post_data から WS満足度・WS理解度を集計し、平均値を算出。既存のスコア変化の記述に**補足として**1〜2文を付加。
- **記載例**: 「ワークショップに対する満足度は平均4.0点、理解度は4.25点と高く、学習内容が適切に伝達されたことがうかがえる。このことから、スコア向上は学習効果によるものと解釈できる。」
- **閾値による言い回し**: 満足度・理解度の平均が3.5以上であれば「高い」、3.0以上3.5未満なら「おおむね良好」、3.0未満なら「一定の課題があり」等、数値に応じて表現を切り替える。

#### 7.3.2 ~~`{{O_block_1_body}}` への組み込み~~ 【廃止】

- `{{O_block_1_body}}` は廃止。組織別のWS理解度の平均はスライド3で **`{{OrkA_1}}`** として表示する（**13_スライド3_組織別分析_プレースホルダー仕様** 参照）。単位「点」なし、小数点0.00まで。

#### 7.3.3 `{{O_block_2_body}}` / `{{O_block_3_body}}` への組み込み（オプション）

- **付加の条件**: その部署の満足度・理解度の平均が3.5以上のときのみ、1文を付加する。
- **O_block_2_body への付加例**: 「参加者の満足度・理解度も高く、強みの現場での定着が期待される。」
- **O_block_3_body への付加例**: 「参加者の学習意欲は高く、弱み領域の改善に向けた取り組みが期待できる。」
- **文字数制限**: 付加は各ブロック50文字程度を目安とし、目標文字数（約300文字）を超えないよう調整する。

### 7.4 実装上のデータフロー

- **入力**: `post_data`（直後.csv 正規化後）。各行に `WS満足度`, `WS理解度`, `NPS`, `所属部署` が含まれる。
- **集計**: 全社レベルでは `analyzer.calculate_satisfaction` または同様のロジックで平均を算出。部署別では `所属部署` でグループ化して平均を算出。
- **既存データの利用**: 04_満足度分析.csv または `generate_satisfaction_analysis_csv` の計算結果を再利用可能。ただし `generate_summary_comment` 等の呼び出し時点で sat_path や satisfaction の辞書が渡されている場合は、それを参照する。
- **変更対象**: `generate_summary_comment_impl` に satisfaction データ（満足度・理解度の平均）を渡す。組織別の理解度はスライド3で `{{OrkA_1}}` として report_generator 等で部署別に算出する（13 参照）。report_generator の generate_slide_content_markdown 内で、satisfaction データを generate_summary_comment に渡す既存の仕組みを拡張する。

### 7.5 満足度・理解度と言語の切り替え

| 平均値 | 満足度・理解度の表現例 |
|--------|------------------------|
| 4.0以上 | 「高い」「非常に高い」 |
| 3.5以上4.0未満 | 「高い」「良好」 |
| 3.0以上3.5未満 | 「おおむね良好」「標準的」 |
| 3.0未満 | 「一定の課題があり」「さらなる工夫の余地がある」 |

### 7.6 Phase 別の適用

- **Phase 1**: 直後.csv が存在しないため、満足度・理解度の活用は行わない。
- **Phase 2**: 上記すべての活用を適用する。
- **Phase 3**: Phase 2 と同様に適用する。

### 7.7 満足度分析スライド（スライド8）との関係

- 満足度・理解度・NPS は**スライド8（満足度分析）**で `{{sc_1}}`, `{{sc_2}}`, `{{sc_3}}`, `{{M_block_1_body}}` として既に詳細に表示されている。
