# 25 エグゼクティブサマリー コメント最適化要件

スライド1「エグゼクティブサマリー」の `{{S_block_1_body}}`・`{{S_block_2_body}}`・`{{S_block_3_body}}` について、**スキル分析の文章に近すぎる**現状を改め、**読み手（特に部長等の役職者）が納得できる**表現に最適化するための要件を規定する。

---

## 1. 背景・課題

### 1.1 対象プレースホルダー

| プレースホルダー | 用途 | データソース |
|----------------|----------|-------------|
| `{{S_block_1_body}}` | 総評コメント | 生成レポート.md（1. エグゼクティブ・サマリー > 総評コメント） |
| `{{S_block_2_body}}` | 強み | 生成レポート.md（1. エグゼクティブ・サマリー > 強み） |
| `{{S_block_3_body}}` | 弱み・伸びしろ | 生成レポート.md（1. エグゼクティブ・サマリー > 弱み） |

### 1.2 現状の課題

- **スキル分析の文章に近い**: 実施前○点→直後○点、〇〇力+0.XXpt、△△力（X.XX点）は… といった**数値・軸名の羅列**が多く、スライド2のスキル分析テーブルや C_block の説明と**役割が重複**している。
- **総合的な総評になっていない**: 総評（S_block_1）が「数字の要約」に留まり、**総合的にどうだったか（傾向）**や**今後どうあるとよいか（方向性）**が弱い。
- **強み・伸びしろが軸別説明になりがち**: S_block_2/3 が「〇〇力（X.XX点）は…」型で、**総合的に何が強みか・何が伸びしろか**が一言で伝わりにくい。部長等が「で、結局どこが強みでどこを伸ばすとよいのか」を掴みづらい。

### 1.3 やりたいこと

- **コメントの文章を最適化**し、読み手が納得できる内容にする。特に**部長など役職者が見た場合に納得感**があるようにする。
- **総評**: 数字より**全体的な傾向**と**今後どうあるとよいか**を書く。
- **強み・伸びしろ**: 数値的な言い方を減らし、**総合的に何が強みか・何が伸びしろか**を表現する。スキル分析と重複しない役割分担にする。

---

## 2. 要件概要

### 2.1 方針

- **スライド1は「経営・マネジメント視点の要約」**として位置づける。スライド2のスキル分析（数値・軸別）はそのまま活かし、スライド1では**解釈・傾向・方向性**を前面に出す。
- **数値は補足**に留め、必要最小限にとどめる。詳細な数値はスライド2の表・レーダーに任せる。
- **納得感**: 根拠（何を根拠に言っているか）は簡潔に示しつつ、**解釈（だから何が言えるか）**と**示唆（だから今後どうするとよいか）**を明確に書く。

---

## 3. プレースホルダー別詳細要件

### 3.1 `{{S_block_1_body}}`（総評コメント）

- **目的**: ワークショップ全体を総合的にどう評価するか、今後どうあるとよいかを、**部長等が一読して納得できる**形で述べる。
- **含めるべき内容**:
  1. **総合的にどうだったか（傾向）**
     - 例: 「ワークショップを通じて、現場で形にすることは得意だが、課題の掘り下げや合意形成に課題があった状態から、バランスの取れたスキル基盤へと変化する兆しが見られた」など、**数値以外の傾向・質的な変化**を一言で表現する。
  2. **学習効果の位置づけ**
     - スコア向上を「学習効果が表れた」と解釈できることを簡潔に。数値は「全体的に向上した」程度に留め、+0.XXpt の羅列は避ける。
  3. **今後どうあるとよいか（方向性）**
     - 例: 「現場での実践意欲が高く、宣言されたアクションが定着すれば、さらに成果につながることが期待される」「継続的な振り返りとチームでの共有が、定着の鍵となる」など、**次の一手・期待**を1〜2文で述べる。
- **避けること**:
  - 実施前○点→直後○点、〇〇力+0.XXpt・△△力+0.XXpt といった**スキル分析と同様の数値列挙**。必要なら「全体的に○pt程度向上」程度に要約する。
  - スライド2の C_block や表と**同じ表現の繰り返し**。
- **文章量**: 約300〜400文字を目安。スライド表示に収まる長さとする。
- **出力元**: `generate_summary_comment_impl`（またはエグゼクティブサマリー専用の総評生成）の出力を、上記方針に沿って**総合総評型**に変更する。

### 3.2 `{{S_block_2_body}}`（強み）

- **目的**: **総合的に何が強みなのか**を、役職者が「なるほど、ここが強みか」と納得できる形で述べる。スキル分析の「〇〇力（X.XX点）は…」の延長にならないようにする。
- **含めるべき内容**:
  1. **強みの本質を一言で**
     - 例: 「現場で素早く形にし、検証を回す力が強みである」「ヒアリングで得た情報を課題の本質に結びつける力が組織の強みとして表れている」など、**軸名・点数より「何が強みか」**を先に書く。
  2. **業務・組織への効き方（解釈）**
     - その強みが、業務や組織にどう効いているか（どう活きているか）を1文で。
  3. **維持・活用の示唆（任意）**
     - 強みを活かす・定着させるための方向性を1文程度。アクション宣言の引用は、強みの「現場での意欲」を示す補足として使う。
- **避けること**:
  - 「具体化・検証力（2.77点）は…」「リサーチ・分析力（2.65点）は…」といった**軸名＋点数から始まる説明**をそのまま並べない。点数は「スコアが相対的に高い」程度の補足にとどめる。
  - スキル分析テーブルや C_block と**同じ言い回し**の重複。
- **文章量**: 約250〜350文字を目安。
- **出力元**: `extend_strength_text_impl` の入力・出力を、**総合強み型**のテンプレート／ロジックに拡張する。またはエグゼクティブサマリー専用の強み文生成を追加する。

### 3.3 `{{S_block_3_body}}`（弱み・伸びしろ）

- **目的**: **総合的に何が伸びしろなのか**、**今後どうあるとよいか**を、役職者が「ここを伸ばすと効果的だ」と納得できる形で述べる。
- **含めるべき内容**:
  1. **伸びしろの本質を一言で**
     - 例: 「計画は立てられるが、チームを巻き込み実行に移すところに伸びしろがある」「コンセプトを明確にし、関係者と合意を取るプロセスに強化の余地がある」など、**軸名・点数より「何が伸びしろか」**を先に書く。
  2. **業務・組織への影響（解釈）**
     - その伸びしろを補うことで、何が良くなるか（どんな成果につながるか）を1文で。
  3. **改善の方向性・示唆**
     - 今後どうあるとよいか（チームでの役割分担、振り返りの仕組み、次の学習テーマなど）を1文程度。参加者のアクション宣言は「改善を志向した意欲」の補足として使う。
- **避けること**:
  - 「実現・ディレクション力（2.48点）は…」「構想・コンセプト力（2.57点）は…」といった**軸名＋点数から始まる説明**の羅列。点数は「スコアが相対的に伸ばせる領域」程度の補足にとどめる。
  - スキル分析と**同じ表現**の重複。
- **文章量**: 約250〜350文字を目安。
- **出力元**: `extend_weakness_text_impl` の出力を、**総合伸びしろ型**のテンプレート／ロジックに拡張する。またはエグゼクティブサマリー専用の伸びしろ文生成を追加する。

---

## 4. 納得感を高めるための共通指針

- **読み手**: 部長・マネジメントなど、数値細部より「傾向と方向性」を重視する役職者を想定する。
- **論理の流れ**: 「（簡潔な根拠）→ 解釈（だから何が言えるか）→ 示唆（だから今後どうするとよいか）」の順で書く。
- **役割分担**:
  - **スライド1（S_block_*）**: 総合的な総評・強み・伸びしろの**本質と方向性**。数値は補足のみ。
  - **スライド2（C_block_*・表）**: スキル軸別の数値・推移・分析。詳細はこちらに任せる。
- **重複の回避**: スライド2ですでに述べている数値列挙や軸別説明を、スライド1で繰り返さない。

## 5. 実装対象とデータフロー

| 対象 | 内容 |
|------|------|
| **総評（S_block_1）** | `report_generator_impl.generate_summary_comment_impl` を拡張するか、エグゼクティブサマリー用の「総合総評」生成関数を追加し、傾向・方向性を前面にした文を返す。数値は要約レベルに抑制。 |
| **強み（S_block_2）** | スライド1用に「総合強み型」の文を生成するロジックを追加。`extend_strength_text_impl` の結果をそのまま使う場合は、先頭に「総合的に～が強みである」のような一文を足す、または S_block 専用のテンプレートで軸名・点数を出さずに本質を書く。 |
| **弱み（S_block_3）** | 同様に「総合伸びしろ型」の文を生成。`extend_weakness_text_impl` の結果を総合伸びしろ型に変換するか、S_block 専用の生成を追加。 |
| **スライド挿入内容・GAS** | 上記変更後、レポート再生成により スライド挿入内容_*.md と GASコード_*.gs に反映される。 |

### 5.1 既存要件との関係

- **10_分析コメント品質要件**: 根拠・解釈・示唆の指針は維持するが、**エグゼクティブサマリーに限り**「数値は補足」「総合的な傾向・方向性を前面に」という本要件を優先する。
- **12_C_block_comment_requirements**: C_block（スライド2）の文章量・内容は従来どおり。S_block との役割分担を本要件で明確化する。

## 6. 受入基準（例）

- **S_block_1**: 「実施前2.29点から直後2.62点へ+0.33pt」のような数値列挙が主になっていない。「総合的に～という傾向が見られ、今後は～するとよい」といった**傾向と方向性**が読める。
- **S_block_2**: 「〇〇力（X.XX点）は…」から始まる文が並んでおらず、**「総合的に何が強みか」**が一文で分かり、部長が「強みはここだ」と納得できる。
- **S_block_3**: 同様に**「総合的に何が伸びしろか」「今後どうあるとよいか」**が分かり、スキル分析の言い回しと重複が少ない。
