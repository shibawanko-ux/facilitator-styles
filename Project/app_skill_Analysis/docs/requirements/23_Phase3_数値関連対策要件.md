# 23 Phase3 数値関連対策要件

Phase 3（1ヶ月後データ含む）において、Phase 2 で発生したような数値の不一致・表示ミスを予防するための要件を規定する。

---

## 1. 背景と目的

- **背景**: Phase 2 で組織別スライドの「値ベース replaceAllText」により、同一数値が複数セルにある場合に誤置換が発生した（業務システム課スコア不一致）。Phase 3 では組織別に **Ogr*_1〜_5**（実施前・直後・1ヶ月後・変化量直後・変化量1ヶ月後）、個人別に **Cgr*_1_P〜_5_P** 等のプレースホルダーが増え、データソースも実施前・直後・**1ヶ月後**の3時点となるため、同様の数値課題が発生しうる。
- **目的**: Phase 3 の数値が正しく計算・表示されるよう、データの一貫性・置換ロジック・検証の要件をあらかじめ定め、実装・運用で守る。

---

## 2. 対象となる数値・データ

| 対象 | プレースホルダー例 | データソース |
|------|-------------------|--------------|
| 組織別スライド | OgrA_1〜OgrF_5（実施前・直後・1ヶ月後・変化量×2） | pre_data / post_data / follow_data を analyze_by_department で集計。02_組織別分析_Phase3.csv は **1ヶ月後** 時点のみ。 |
| 個人別スライド | CgrA_1_P〜CgrF_5_P、Ggr*（ギャップ） | 実施前・直後・1ヶ月後 CSV と個人マッチング。 |
| 全社スライド | Cgr*_1〜_5（1ヶ月後・変化量1ヶ月後） | 01_エグゼクティブサマリー等。 |

---

## 3. 要件一覧

### R1: 組織別スライドの置換ロジック（Phase 2 と同じ方針）

- GAS の組織別スライド生成では、**値ベースの replaceAllText(oldValue, newValue)** を用いない。
- **全組織で「プレースホルダーが残ったテンプレート」を複製し、複製にのみ「プレースホルダー → その組織の値」で replacePlaceholdersInSlide_(slide, orgDict) を行う**（14・組織別スライド_スコア不一致修正メモで Phase 2 対応済み。Phase 3 用 GAS も同一の gas_generator から生成するため、同じロジックが適用される）。
- Phase 3 で Ogr*_3, Ogr*_4, Ogr*_5 が増えても、**プレースホルダーキー（例: `{{OgrB_3}}`）で置換する**ため、異なるセルに同じ数値が入っていても誤置換しない。

### R2: 組織別のデータソースと 02_組織別分析_Phase3.csv の役割

- **02_組織別分析_Phase3.csv** は、**1ヶ月後（follow_data）** を analyze_by_department した結果のみを出力する（現行の generate_department_analysis_csv の仕様のまま。1時点のみ）。
- スライド挿入内容・GAS の ORGANIZATION_DATA で必要な **実施前・直後・1ヶ月後・変化量** は、次のように算出する。
  - **実施前**: analyze_by_department(pre_data) の該当部署
  - **直後**: analyze_by_department(post_data) の該当部署
  - **1ヶ月後**: analyze_by_department(follow_data) の該当部署（＝ 02_組織別分析_Phase3.csv と一致）
  - **変化量(直後)**: 直後 − 実施前
  - **変化量(1ヶ月後)**: 1ヶ月後 − 直後（または 1ヶ月後 − 実施前。仕様で明示する）
- スライド挿入内容 MD の組織別セクションでは、**Phase 3 のとき** Ogr*_3（1ヶ月後）・Ogr*_5（変化量1ヶ月後）に **follow_data から算出した値を出力**すること。Phase 2 と同様に「-」固定にしない。

### R3: 所属部署の整合性（3時点）

- 組織別集計で使う「所属」は、**実施前・直後・1ヶ月後** の各 CSV で **同一のキー（strip 済み）** で揃える必要がある（[21_組織別スコアと所属部署の整合性](21_組織別スコアと所属部署の整合性.md) を Phase 3 に拡張）。
- 1ヶ月後.csv の「所属部署」列も、実施前・直後と **同じ選択肢・同じ表記** にすること。正規化は csv_normalizer で strip 等を適用済みとする。
- 所属が直後と1ヶ月後で異なる参加者がいる場合、組織別の「人数」や「1ヶ月後スコア」の解釈をどうするか（1ヶ月後時点の所属で集計するか、直後時点の所属で集計するか）を仕様で決め、実装とドキュメントを一致させる。

### R4: Phase 3 用の組織別スライド挿入内容（Ogr*_3, Ogr*_5 の出力）

- report_generator の generate_slide_content_markdown において、**phase == 3 かつ follow_data が存在する場合**、組織別の「スキル分析テーブル（表内）」で次を満たすこと。
  - Ogr*_3: 当該組織の **1ヶ月後** スコア（analyze_by_department(follow_data) から取得）を記載する。
  - Ogr*_5: 当該組織の **変化量(1ヶ月後)**（1ヶ月後 − 直後、または仕様で定めた計算式）を記載する。
  - Ogr*_4 は Phase 2 と同様に変化量(直後)。
- Phase 2 のときは従来どおり Ogr*_3 / Ogr*_5 を「-」または「Phase2のため未使用」で出力してよい。

### R5: 個人別 Phase 3 の数値

- 個人別スライドの Cgr*_3_P（1ヶ月後）・Cgr*_5_P（変化量1ヶ月後）等は、**プレースホルダー置換のみ**で埋める（individual_gas_generator はスライド挿入内容（個別）.md の表から placeholders を収集し、replacePlaceholdersInSlide_ で置換するのみ。値ベースの一括置換は行わない）。
- スライド挿入内容（個別）.md は、Phase 3 の場合に 1ヶ月後・変化量(1ヶ月後) の列を正しく出力すること。データソースは follow_data と個人マッチング（メールアドレス）の結果。

### R6: Phase 3 用の検証

- **組織別**: 実施前・直後・1ヶ月後の3時点について、analyze_by_department の再計算値と、スライド挿入内容 MD（および 02_組織別分析_Phase3.csv の1ヶ月後部分）が一致することを検証できるようにする。
  - 既存の verify_organization_scores.py は Phase 2 前提（02_組織別分析_Phase2.csv と post_data）のため、**Phase 3 対応オプション**（例: --phase 3、または 1ヶ月後 CSV と 02_組織別分析_Phase3.csv を指定）を追加し、Ogr*_1〜_5 の期待値とスライド挿入内容・02 CSV の一致を確認できるようにする。
- **個人別**: Phase 3 時は 1ヶ月後・変化量(1ヶ月後) が生成レポート（個別）.md およびスライド挿入内容（個別）.md に含まれることを確認する。必要に応じて、個別レポートの数値とスライド挿入内容の Cgr*_3_P, Cgr*_5_P の一致を検証するスクリプトや手順を検討する。

### R7: GAS 生成元の統一

- Phase 3 用の GAS コードも **gas_generator.py** から生成する。組織別スライドの「テンプレート複製＋プレースホルダー置換のみ」のロジックは Phase に依存せず共通であるため、Phase 3 用 .gs も同じテンプレート・同じ buildOrgSlidesFromCSV_ を用いる。Phase 3 専用の値ベース置換を追加しない。

---

## 4. 実装・運用上の注意

- **02_組織別分析_Phase3.csv**: 現状は「1ヶ月後」1時点のみ。スライドで実施前・直後を表示する場合は、CSV を拡張するのではなく、スライド挿入内容生成時に pre_data / post_data を都度 analyze_by_department して Ogr*_1, Ogr*_2 等に充てる（13 の「02は拡張しない」に準拠）。
- **検証の習慣**: Phase 3 プロジェクトでレポート・GAS を再生成したあと、可能であれば Phase 3 用検証（R6）を実行し、組織別・必要に応じて個人別の数値一致を確認する。
- **ドキュメント**: 本要件は 21（組織別スコアと所属部署の整合性）・13（組織別分析プレースホルダー）・14（組織別スライド表プレースホルダー置換）を Phase 3 に拡張するものとして参照する。
