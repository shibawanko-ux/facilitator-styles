# 05 分析と出力仕様

## 1. 分析ロジック（実装の正）

### 1.1 軸スコアの計算

- 各軸スコア = その軸に属する**3問の平均**（欠損は除く）。
- スコア有効範囲: **1〜5**。範囲外は無視する。

### 1.2 レーダーチャートの軸順

- 表示順は評価軸の定義順と同じ: リサーチ・分析力 → 構想・コンセプト力 → 具体化・検証力 → 伝達・構造化力 → 実現・ディレクション力。
- 5角形のレーダーチャートで組織（または個人）のスコアを可視化する。

### 1.3 個人別の表示名（参加者名）

- 個別レポート・スライド挿入内容・個別 GAS・エクスポート CSV で使う参加者名は、**実施前 → 直後 → 1ヶ月後** の各 CSV の「氏名」を優先し、いずれも空のときのみメールアドレスの @ より前を用いる。
- 詳細は **08_表示名（参加者名）仕様** を参照。

### 1.4 メールアドレス整合性検証（Phase 2 以上）

- Phase 2 以上で分析を実行する前に、実施前・直後のメールアドレス整合性を検証する。
- **実施前のメール重複**: 同じメールアドレスが複数の氏名で登録されている場合はエラー。
- **直後のメール重複**: 同上。
- **実施前と直後の不一致**: 実施前にいる人のメールアドレスが直後に1件もマッチしない場合はエラー。
- 検証に失敗した場合は分析を実行せず、画面上にエラーメッセージを表示する。実装は `src/analyzer.py` の `validate_email_consistency`。

### 1.5 分析の種類

- **全社スキル・レーダーチャート**: 組織全体の5軸スコア。
- **組織別・属性別比較**: 所属部署等でグルーピングした比較。
- **定着度・ギャップ分析**（Phase 3）: 本人評価と実践頻度・上長評価のクロス分析。「知識止まり」層の特定に利用する。

### 1.5.1 ギャップ計算式（確定仕様）

```
ギャップ = 本人(1ヶ月後) − 上長
```

| ギャップ値 | 意味 | 読み方 |
|-----------|------|--------|
| **正（+）** | 本人が上長より高く自己評価している | 過信の可能性。上長は実践の質を低く見ている |
| **0** | 認識一致 | 本人と上長の評価が揃っている |
| **負（−）** | 上長が本人より高く評価している | 本人が謙遜しすぎている。上長はより高い実践を認めている |

- 実装: `src/analyzer.py` の `calculate_gap(self_scores, manager_scores)` → `self_scores[key] - manager_scores[key]`
- `self_scores` は **1ヶ月後（follow）** の実スコア。変化量（差分）は使わない。

### 1.6 強み・弱みの判定ロジック

**強み**（S_block_2_body / O_block_2_body）:
- 5軸のうち、**平均スコアより高い**スキル軸を強みとして特定する（最大3件、スコア降順）。
- 平均より高いスキルが1件もない場合は、**最高スコアのスキルを1件**強みとして返す（相対的に優れている領域を示す）。
- 0以下のスコアは除外する。

**弱み**（S_block_3_body / O_block_3_body）:
- 5軸のうち、**平均スコアより低い**スキル軸を弱みとして特定する（最大2件、スコア昇順）。
- 平均より低いスキルが1件もない場合は、**最低スコアのスキルを1件**弱みとして返す。
- 0以下のスコアは除外する。

※ 絶対閾値（例: 3.5点以上）は用いず、組織・部署内の相対比較で強み・弱みを判定する。

## 2. 生成するレポート・ファイル

**Phase 1（実施前のみ）**: GAS 用の出力は行わない。Phase 1 だけに関する専用アウトプット（GAS およびそれのみの成果物）は**必須としない**（実装しなくてもよい）。Phase 2/3 からレポート・CSV・GAS 連携を提供すればよい。

### 2.1 Markdown 分析レポート

- エグゼクティブサマリー
- 全社スキル分析
- 強み・弱み分析
- Phase 3 の場合: 定着度・ギャップ分析、推奨プログラム提案

**生成レポート関連の確認用構成**: 生成レポート（全体）・生成レポート（個別）・スライド挿入内容（全体・個別）の構成とデータソース表記を整理し、確認者が「1枚のレポート情報」として照合できるようにする要件は **18_生成レポート統合確認と構成整理要件** を参照する。

### 2.2 レーダーチャート

- PNG 画像。5軸のスコアを可視化する。（Phase 2/3 で生成すればよい。Phase 1 では必須としない。）

**格納先（分析ボタン実行時）**

- **全体用レーダーチャート**: 従来どおり、プロジェクトの `reports/` 直下に格納する。
  - 例: `生成レポート_{プロジェクト名}_Phase{phase}_レーダーチャート.png`
- **個人用レーダーチャート（Phase 2）**: プロジェクトの `reports/Phase2_personal_radar_chart/` に格納する。
  - 例: `reports/Phase2_personal_radar_chart/生成レポート（個別）_{プロジェクト名}_{参加者名}_Phase2_レーダーチャート.png`
- **個人用レーダーチャート（Phase 3）**: プロジェクトの `reports/Phase3_personal_radar_chart/` に格納する。
  - 例: `reports/Phase3_personal_radar_chart/生成レポート（個別）_{プロジェクト名}_{参加者名}_Phase3_レーダーチャート.png`

分析実行時に、上記 Phase 別フォルダが存在しなければ自動作成する。生成レポート（個別）.md および スライド挿入内容（個別）.md 内のレーダーチャート画像参照は、`reports` を基準とした相対パス（例: `Phase2_personal_radar_chart/生成レポート（個別）_...png`）で記載する。

### 2.4 スライド挿入内容（個別）（Phase 2 以上）

- 個人別スライド用のデータを記載した **スライド挿入内容（個別）_{プロジェクト名}_Phase{N}.md** を生成する。
- 各参加者セクションに **活用意欲・アクション宣言** を記載する。プレースホルダー `{{C_block_3_1_body_P}}`（活用意欲）・`{{C_block_3_2_body_P}}`（アクション宣言）に対応するデータ内容は、直後.csv の Q16A（入力文言で表示）・Q17A（自由記述）から取得する。詳細は **06_GAS連携** の 5.1 を参照。

### 2.5 スライド挿入内容（全体用）のプレースホルダー形式

- スライド2（スキル分析）の**スキル分析テーブル（表内）**に総合スコア行を追加する場合、`{{CgrF_1}}`（実施前）・`{{CgrF_2}}`（直後）・`{{CgrF_4}}`（変化量(直後)）等を用いる。詳細は **17_スキル分析テーブル_総合スコア行追加要件** を参照。
- 組織別分析スライド（LAYOUT: **analysis_by_organization**）のプレースホルダー・表記ルールは **13_スライド3_組織別分析_プレースホルダー仕様** を参照。
- 組織別分析スライドの `{{O_score}}`（`{{O_overall_score}}` 同義）: 部署の総合スコアを**数値のみ**で出力する。単位「点」は付けない。
  - 例: `2.62`（`2.62点` ではなく）
- 分析コメントの文章量・納得度: **10_分析コメント品質要件**、**12_C_block_comment_requirements** を参照。
  - `{{S_block_1_body}}`・`{{C_block_1_body}}`: 約400文字
  - `{{O_block_2_body}}`・`{{O_block_3_body}}`・`{{C_block_2_1_body}}`・`{{C_block_2_2_body}}`: 約300文字
  - ※ `{{O_block_1_body}}` は廃止（13 参照）。

## 3. 推奨プログラム選定ロジック

分析で明らかになった**弱点領域（5軸のうちスコアが相対的に低い軸）**に応じて、補強用のワークショッププログラムを提案する。課題領域と推奨プログラムの対応は、評価軸定義と整合させる（例: リサーチ・分析力が弱い → ユーザーリサーチ／インタビュー系プログラム）。詳細は参照元の 05_推奨プログラム詳細 を参照する。
