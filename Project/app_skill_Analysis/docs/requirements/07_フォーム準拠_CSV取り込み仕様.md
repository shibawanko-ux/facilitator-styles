# 07 フォーム準拠 CSV 取り込み仕様

Google フォーム等からエクスポートされた CSV を**そのまま**アップロードして利用できるようにするための仕様です。フォームの「質問文が列名になる」「回答が日本語ラベルになる」ことをアプリ側で吸収し、既存の分析・レポート・GAS 連携ロジックは変更しません。

---

## 1. 目的とスコープ

### 1.1 目的

- **フォーム構成を正とする**: ユーザーがスプレッドシートや GAS で列名・値を手作業で編集しなくても、フォームからダウンロードした CSV をそのままアップロードして分析できるようにする。
- **既存ロジックはそのまま**: 分析・レポート・GAS 生成は 01〜06 の要件および既存の `analyzer` / `report_generator` の前提（短いカラム名・1〜5 の数値）を変えず、**読み込み直後にだけ**「列名の正規化」と「値の正規化」を行う。

### 1.2 スコープ

| 対象 | 内容 |
|------|------|
| 実施前.csv | 列名マッピング、Q1〜Q15 の値マッピング（日本語ラベル → 1〜5） |
| 直後.csv | 上記に加え、WS満足度・WS理解度・NPS・Q16A・Q17A の列名・値マッピング |
| 1ヶ月後.csv | 実施前と同様の Q1〜Q15 ＋ Q16B・Q17B の列名・値マッピング |
| 上長1ヶ月後.csv | 対象者メールアドレス、M1〜M5 等の列名・値マッピング |
| 既存の短い列名・数値の CSV | 正規化後も変化しないようにし、従来どおり動作する |

---

## 2. 処理フロー

```
[アップロード CSV] → [文字コード・BOM 処理] → [列名正規化] → [値正規化] → [既存 analyzer / report_generator]
```

- **列名正規化**: CSV の 1 行目（ヘッダー）を、内部で使うカラム名（`メールアドレス`、`Q1`〜`Q15`、`WS満足度` 等）に変換する。フォームの質問文と「完全一致」または「先頭から含む」でマッピングする。
- **値正規化**: スコア列（Q1〜Q15、WS満足度、WS理解度、Q16A、Q16B、NPS、M1〜M5）について、日本語ラベルを 1〜5（NPS は 0〜10、Q16B は 1〜4）の数値に変換する。もともと数値の場合はそのまま通過させる。
- 正規化は **CSV を読み込んだ直後**にのみ行い、以降のモジュールには常に「短いカラム名・数値」の辞書リストを渡す。

---

## 3. 列名マッピング

### 3.1 実施前・直後・1ヶ月後（参加者用）CSV

フォームの 1 行目が「質問文そのまま」の場合、以下のいずれかに一致または**先頭から含む**列名を、対応する内部カラム名に変換する。複数候補がある列は、先にマッチしたものを採用する。

| 内部カラム名 | マッピング候補（フォーム側の列名の例） |
|--------------|----------------------------------------|
| `メールアドレス` | `メールアドレス`、`あなたのメールアドレスを入力してください。`、先頭が「メール」で始まる列 等 |
| `氏名` | `氏名`、`お名前`、`名前`、`あなたの名前を入力してください。`、`あなたの氏名を入力してください。` 等（表示名の決定ロジックは **08_表示名（参加者名）仕様** を参照） |
| `所属部署` | `所属部署`、`あなたの所属部署を選択してください。`、`所属`、BOM付き`所属部署` 等。**運用**: 実施前・直後で所属の**値**が同一になるようフォームの選択肢を揃えること。表記ゆれがあると組織別実施前スコアが取れない場合がある（21_組織別スコアと所属部署の整合性 参照）。 |
| `Q1` | `Q1`、`Q1：` で始まる列、`Q1: 仮説検証の設計`、`解決すべき課題に対し…` 等（Q1 に相当する質問文） |
| `Q2` 〜 `Q15` | 同様に `Q2`〜`Q15`、または「Q2: インサイトの発掘」等の質問文 |
| `WS満足度` | `WS満足度`、`満足度`、`本ワークショップの満足度`、`今回のワークショップの内容に対する満足度はどの程度ですか？` 等 |
| `WS理解度` | `WS理解度`、`理解度`、`本ワークショップの内容の理解度`、`今回のワークショップの内容をどの程度理解できましたか？` 等 |
| `NPS` または `NPS(推奨度)` | `NPS`、`NPS(推奨度)`、`推奨度`、`知人に勧めたい度`、`このワークショップを、同僚や知人にどの程度おすすめしたいですか？` 等 |
| `Q16A` | `Q16A`、`Q16A:` / `Q16A：`、`活用意欲`、`Q16A：今回学んだ内容を、明日からの業務で活用できるイメージが湧いていますか？` 等 |
| `Q17A` | `Q17A`、`Q17A:` / `Q17A：`、`アクション宣言`、`Q17A：学んだ内容を踏まえ、直近で「変えてみよう」「やってみよう」と思う具体的な行動を1つ宣言してください。` 等 |
| `Q16B` | `Q16B`、`Q16B: 実践頻度`、`実践頻度` 等 |
| `Q17B` | `Q17B`、`Q17B: 実践エビデンス`、`実践エピソード`、`コメント`、`自由記述` 等 |

**マッピング順序（実装上の必須事項）**: 列名は「先頭から含む」でマッチするため、`Q16A：…` や `Q17A：…` のような列名が **Q1** の候補 `Q1` に誤ってマッチしないよう、**Q16A・Q17A は Q1 より前に評価する**順序でマッピング表を定義すること。実装は `src/csv_normalizer.py` の `COLUMN_MAP_PARTICIPANT` で Q16A・Q17A を Q2 の直後（Q1 の前）に配置している。

- **Q1〜Q15**: フォームの質問順序が 02_評価フレームワーク と一致している前提で、「1 列目から順に Q1, Q2, …」でマッピングする方式、または「列名に Q1〜Q15 が含まれる」方式のいずれか（あるいは両方）を実装する。質問文のみで列名に Q 番号がない場合は、**列の並び順**で Q1〜Q15 に対応づける定義を別表（フォーム別マッピング表）で持つことを許容する。

**直後フォームの満足度関連3問**: 直後フォームでよく使われる以下の3問は、列名が質問文のままエクスポートされても上表の内部カラム名にマッピングする。実装は `src/csv_normalizer.py` の `COLUMN_MAP_PARTICIPANT` および値マッピング（4.2〜4.6）に従う。
- 満足度: 「今回のワークショップの内容に対する満足度はどの程度ですか？」 → `WS満足度`
- 理解度: 「今回のワークショップの内容をどの程度理解できましたか？」 → `WS理解度`
- 推奨度（NPS）: 「このワークショップを、同僚や知人にどの程度おすすめしたいですか？…」 → `NPS`

### 3.2 上長1ヶ月後.csv

| 内部カラム名 | マッピング候補（フォーム側の列名の例） |
|--------------|----------------------------------------|
| `対象者メールアドレス` | `対象者メールアドレス`、`評価対象者のメールアドレス`、`本人のメールアドレス` 等 |
| `上長メールアドレス` / `メールアドレス` | `上長メールアドレス`、`メールアドレス`、`回答者メールアドレス` 等 |
| `上長氏名` / `氏名` | `上長氏名`、`氏名`、`お名前` 等 |
| `上長部署` / `所属部署` | `上長部署`、`所属部署` 等 |
| `M1` 〜 `M5` | `M1`〜`M5`、`M1: リサーチ・分析力（上長評価）` 等 |
| `M7` / `上長コメント` / `コメント` | `M7`、`上長コメント`、`コメント`、`自由記述` 等 |

## 4. 値マッピング（日本語ラベル → 数値）

スコア列のセル値が日本語ラベルの場合、以下のルールで数値に変換する。**既に 1〜5（または 0〜10、1〜4）の数値または数値文字列の場合は変換しない**。

### 4.1 Q1〜Q15（5段階: 1〜5）

フォームでよく使われるラベルの例と、数値の対応。実装時は下表を「ラベル → 数値」の辞書として持ち、前後空白は trim してからマッチングする。

| 数値 | ラベル例（1が低い・5が高い） |
|------|------------------------------|
| 1 | 全くできない、できない、そう思わない、当てはまらない 等（02_評価フレームワークの1点に相当） |
| 2 | あまりできない、あまりそう思わない、助けがあればできる 等（同2点） |
| 3 | どちらともいえない、一人でできる、ある程度できる、そう思う 等（同3点） |
| 4 | 工夫・応用してできる 等（同4点） |
| 5 | できる、とてもそう思う、当てはまる、人に教えられる 等（同5点） |

- 上表の数値とラベルの対応は **02_評価フレームワーク** の5段階定義（1=できない／知らない、2=助けがあればできる、3=一人でできる、4=工夫・応用してできる、5=人に教えられる）に合わせる。クライアント別のラベルセットが必要な場合は、`src/csv_normalizer.py` のマッピング表を拡張する。

### 4.2 WS満足度（5段階: 1〜5）

直後フォームの「今回のワークショップの内容に対する満足度はどの程度ですか？」に対応する。選択肢は以下にマッピングする。

| 数値 | ラベル例 |
|------|----------|
| 5 | 大変満足、とても満足 |
| 4 | 満足 |
| 3 | どちらともいえない |
| 2 | 不満、やや不満 |
| 1 | 大変不満、とても不満 |

### 4.3 WS理解度（5段階: 1〜5）

直後フォームの「今回のワークショップの内容をどの程度理解できましたか？」に対応する。選択肢は以下にマッピングする。

| 数値 | ラベル例 |
|------|----------|
| 5 | 理解できた、よく理解できた、深く理解できた |
| 4 | ある程度理解できた |
| 3 | どちらともいえない、どちらでもない |
| 2 | あまり理解できなかった |
| 1 | 理解できなかった、全く理解できなかった |

### 4.4 Q16A（活用意欲・5段階: 1〜5）

| 数値 | ラベル例 |
|------|----------|
| 5 | 湧いている、とても湧いている |
| 4 | やや湧いている |
| 3 | どちらともいえない |
| 2 | あまり湧いていない |
| 1 | 湧いていない、全く湧いていない |

### 4.5 Q16B（実践頻度・4段階: 1〜4）

| 数値 | ラベル例 |
|------|----------|
| 1 | よくあった、週1回以上 |
| 2 | たまにあった、月数回程度 |
| 3 | ほとんどなかった、1回程度 |
| 4 | 全くなかった |

### 4.6 NPS（0〜10）

- 直後フォームの「このワークショップを、同僚や知人にどの程度おすすめしたいですか？（0点: 全くおすすめしない 〜 10点: 強くおすすめする）」に対応する。多くの場合フォームでは 1〜10 または 0〜10 の数値で出力される。数値の場合はそのまま。ラベルのみの場合は「0」〜「10」の文字列にマッピングする。

### 4.7 M1〜M5（上長評価・5段階: 1〜5）

- Q1〜Q15 と同様の 5 段階ラベル → 1〜5 のマッピングを適用する。

---

## 5. エンコーディング・BOM

- 入力 CSV の文字コードは **UTF-8** を想定する。
- **BOM 付き UTF-8** の場合、読み込み時に BOM を除去して扱う（ヘッダー先頭列の `\ufeff` 等を削除）。これにより「BOM付き`所属部署`」が正しく `所属部署` として認識されるようにする。

---

## 6. 既存 CSV との互換性

- すでに短いカラム名（`Q1`、`メールアドレス` 等）と数値だけの CSV を使っている場合は、**列名正規化で変化せず、値正規化でも数値はそのまま通過**させる。
- 正規化処理は「フォーム由来かどうか」を区別せず、**すべてのアップロード CSV に一律で適用**してよい。その結果、従来の CSV も同じパスで処理され、挙動は変わらない。

---

## 7. 実装上の配置・責務

| 責務 | 推奨配置 |
|------|----------|
| CSV 読込（ファイル → 辞書リスト） | 既存の `project_manager` またはアップロード処理 |
| 列名正規化・値正規化 | 新規モジュール（例: `csv_normalizer.py`）または `project_manager` 内の「読込直後」の 1 関数群 |
| 正規化のマッピング表（列名・ラベル→数値） | 設定ファイル（YAML/JSON）または定数モジュールで一元管理し、フォーム文言の変更時に修正しやすくする |

- `analyzer.py` / `report_generator.py` / `gas_generator.py` 等は、**正規化済みのデータ**だけを受け取る前提のまま変更しない。

---

## 8. フォーム構成を変更する場合の影響

フォームの「質問の意味」や「文言」を変えても、**CSV のカラム名とスコアの範囲（Q1〜Q15: 1〜5、M1〜M5: 1〜5、NPS: 0〜10）**が変わらなければ、アプリの修正は**不要**です。逆に、**カラム名・質問番号・軸の対応・スコア範囲**を変えると、下記モジュールの修正が必要になります。

| 対象 | 主な依存カラム |
|------|----------------|
| スコア計算・Phase 判定 | メールアドレス、Q1〜Q15、所属部署、Q16B、M1〜M5、対象者メールアドレス |
| **メールアドレス整合性検証**（Phase 2 以上） | 実施前・直後の `メールアドレス` が一貫していること。同一参加者は実施前・直後で同じメールアドレスを使用すること。重複（同一メールが複数氏名で登録）や不一致（実施前にいる人が直後にマッチしない）があると分析が実行されずエラーを表示する。実装は `src/analyzer.py` の `validate_email_consistency`。 |
| 満足度・実践頻度 | WS満足度、WS理解度、NPS、Q16B |
| 部署別分析 | 所属部署、Q1〜Q15 |
| 本人上長比較・ギャップ | メールアドレス、対象者メールアドレス、Q1〜Q15、M1〜M5、M7 等 |
| レポート・CSV・GAS | 上記すべて＋Q17A、Q17B |

**変更パターン別**: 所属部署の**選択肢（値）**を変える → 影響なし。**質問文（ラベル）だけ**変える → 影響なし（列名が同じなら）。**Q1〜Q15 の順序**を変える → 大（02_評価フレームワーク と `src/analyzer.py` の SKILL_AXES を同時に変更）。**列名が質問文のまま**になる → スプレッドシートで列名を編集するか、アプリのマッピング（本 07 の案C）で吸収。**Q1〜Q15 の数・軸の再編** → 大（02・04・analyzer・report_generator・GAS 一式）。**M1〜M5 の順序・意味**変更 → 大（analyzer の軸対応）。**スコア段階**（1〜5 以外）→ 中〜大。**個人識別子**をメール以外に → 大。

## 9. 他ドキュメントとの関係

- **04_データ仕様とフェーズ.md**: 正規化後のデータが満たすべき「必須カラム」「Q1〜Q15 は 1〜5」等の仕様は 04 のとおり。本 07 は「フォームから得られる生 CSV を 04 の形に変換する」ための仕様である。
- フォームの質問順・軸対応（Q1〜Q15 の並び）を変える場合は、02_評価フレームワーク と `src/analyzer.py` の見直しが必要。

---

## 10. まとめ

| 項目 | 内容 |
|------|------|
| 目的 | フォームエクスポート CSV をそのまま使えるようにする |
| 手段 | 読込直後の「列名正規化」と「値正規化」のみ追加 |
| 列名 | 質問文・長い列名を 04 で定める短いカラム名にマッピング |
| 値 | 日本語ラベルを 1〜5（NPS は 0〜10、Q16B は 1〜4）に変換 |
| 既存 | 短い列名・数値の CSV は正規化後も変化させず互換を保つ |

**実装状況**: 列名・値マッピングは `src/csv_normalizer.py` に実装済み。CSV 読込直後に `app.py` で正規化を適用し、以降の analyzer / report_generator には正規化済みデータのみが渡される。
