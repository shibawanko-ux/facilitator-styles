# システムコーチング チーム状態可視化アプリ 要件定義

## 1. プロジェクト概要

### 目的
システムコーチングの実施前後におけるチームの関係性・状態をアンケートによって定点可視化する。チームの健康状態を時系列で追跡し、強み・弱み・推奨アクションを提示する。

### 参考サービス
- リンクアンドモチベーションのチーム状態アプリ（アンケートによる可視化）
- Google Analytics（時系列グラフによるデータ表示）

### 理論的背景
- システムコーチング（CRR Global）
- コルブの経験学習モデル
- ジョン・デューイの組織開発論
- Amy Edmondson（心理的安全性）
- Karen A. Jehn（コンフリクト理論）

---

## 2. ユーザー定義

| ロール | 説明 | 権限 |
|--------|------|------|
| アドミン（ファシリテーター） | アプリ全体を管理する | 全チームの閲覧・管理・アンケート配信 |
| チームリーダー | チームを管理する | 指定された複数チームの閲覧 |
| チームメンバー | アンケートに回答する | 自分のチームのみ閲覧・回答 |

---

## 3. 機能要件

### 3-1. チーム管理
- 複数チームを登録・管理できる
- チームメンバーを招待・追加・削除できる
- メンバーは自分のチームのみ閲覧できる
- リーダーは権限付与により複数チームを閲覧できる

### 3-2. アンケート管理
- アドミンが任意のタイミングで手動配信する
- 事前アンケート・事後アンケートを区別して配信できる
- アンケートの回答状況（回答済み・未回答）を確認できる

### 3-3. アンケート回答
- チームメンバーがWebブラウザ・スマホで回答できる
- 回答形式は6段階評価＋自由記述
- 回答は匿名で集計される

### 3-4. 結果の可視化
- **レーダーチャート**：現時点の6軸バランスを表示
- **折れ線グラフ**：アンケート実施ごとの総合点の時系列変化を表示
- **診断結果**：強み・弱み・今後の推奨アクションを表示
- 自由記述はサマリーとして表示する（AI分析）

## 4. 評価軸（6軸）

| 軸 | 説明 | 理論的背景 |
|----|------|-----------|
| Psychological Safety（心理的安全性） | 安心して発言・失敗共有できるか | Amy Edmondson |
| Voice（声の多様性） | 誰の声が出ているか、少数派の声が尊重されているか | システムコーチング |
| Conflict Quality（対立の質） | 対立が健全（タスク）か破壊的（リレーション）か | Karen A. Jehn |
| Alignment（目的共有） | チームが同じ方向を向いているか | |
| Relationship Quality（信頼） | メンバー間の信頼・相互尊重の質 | |
| System Awareness（システム意識） | 「私vsあなた」から「私たち」への意識変化 | システムコーチング |

## 5. 質問設計

### 回答形式（6段階）
| 1 | 2 | 3 | 4 | 5 | 6 |
|---|---|---|---|---|---|
| 全くあてはまらない | あてはまらない | どちらかといえばあてはまらない | どちらかといえばあてはまる | あてはまる | 非常にあてはまる |

### 質問一覧（共通：事前・事後）18問

**Alignment（目的共有）**
- Q1：チームが向かう方向を、メンバー全員で共有できている
- Q2：チームの目標に対して、自分ごととして取り組めている
- Q3：成果が出ないとき、チーム全員で原因を考えようとする

**Relationship Quality（信頼）**
- Q4：チームメンバーに、本音で話せる関係がある
- Q5：困ったときに、チームメンバーに頼ることができる
- Q6：メンバーそれぞれの強みや役割を、お互いに理解している

**Conflict Quality（対立の質）**
- Q7：仕事の進め方について、率直に意見を言い合える
- Q8：チーム内に、個人間のわだかまりや緊張感がある ※逆転スコア
- Q9：意見の違いが、良い議論やアイデアにつながっている

**System Awareness（システム意識）**
- Q10：自分の仕事がチーム全体にどう影響するか、意識して動いている
- Q11：「私vsあなた」ではなく「私たちの問題」として考えられている
- Q12：チームの一員であることに、意味ややりがいを感じている

**Voice（声の多様性）**
- Q13：チームでは、さまざまなメンバーが発言できている
- Q14：少数派や普段静かなメンバーの意見も、大切にされている
- Q15：自分の意見を、安心してチームで伝えられる

**Psychological Safety（心理的安全性）**
- Q16：ミスや失敗を、チームで正直に話せる雰囲気がある
- Q17：新しいアイデアを出しても、否定されない安心感がある
- Q18：「おかしい」と思ったことを、声に上げられる

### 事前アンケートのみ（自由記述）
- Q19：このチームに期待していることを教えてください
- Q20：チームとしての課題や気になっていることはありますか？

### 事後アンケートのみ（自由記述）
- Q21：セッションを通じて、どんな変化を感じましたか？
- Q22：今後のチームに向けて、一言お願いします

## 6. 認証フロー

```
1. アドミンがメンバーのメールアドレス＋チームを登録
2. メンバーに招待メールが自動送信される
3. メンバーがリンクをクリック → /auth/callback → パスワード設定画面（/set-password）に自動遷移
4. メンバーがパスワードを自分で設定
5. 以降はメールアドレス＋パスワードでログイン
```

### リダイレクト仕様
- 招待リンク・パスワードリセットリンクは `/auth/callback` を経由する
- `/auth/callback` でURLのハッシュパラメータ（`type=invite` / `type=recovery`）を判定し、パスワード設定画面に自動遷移する
- Supabase URL Configuration の Redirect URLs に `http://localhost:5174/auth/callback` を登録する

## 9. アンケート実施仕様

### 実施タイミング
```
プロジェクト
├── 第1回実施前アンケート　← 共通質問＋事前専用質問
├── 第1回実施後アンケート　← 共通質問
├── 第2回実施前アンケート　← 共通質問＋事前専用質問
├── 第2回実施後アンケート　← 共通質問
├── 第N回実施前アンケート　← 共通質問＋事前専用質問
└── 第N回実施後アンケート　← 共通質問＋事後専用質問（最終回フラグをアドミンが指定）
```

### 回数
- 実施回数はクライアントと相談して決める（可変）
- アドミンが最終回フラグを指定することで事後専用質問（NPS等）が表示される

## 11. スコア計算ロジック

### 逆転スコア処理
Conflict Quality（対立の質）のリレーションコンフリクト項目は逆転処理を行う。
- 対象：Q12（人間関係の緊張）・Q13（感情的な対立）
- 計算式：逆転スコア = 7 - 元のスコア（6段階の場合）

### 各軸スコアの計算
各軸に属する質問の平均スコアを算出する。

### 重み付け

| 軸 | 重み | 理由 |
|---|------|------|
| System Awareness（システム意識） | 1.5 | システムコーチングの核心 |
| Psychological Safety（心理的安全性） | 1.3 | 他の軸が機能するための前提条件 |
| Voice（声の多様性） | 1.2 | システムコーチング特有の重要指標 |
| Alignment（目的共有） | 1.0 | 標準 |
| Relationship Quality（信頼） | 1.0 | 標準 |
| Conflict Quality（対立の質） | 1.0 | 標準 |

### 総合点の計算式

```
総合点 =
  (System Awareness × 1.5
  + Psychological Safety × 1.3
  + Voice × 1.2
  + Alignment × 1.0
  + Relationship Quality × 1.0
  + Conflict Quality × 1.0)
  ÷ 7.0（重みの合計）
  × 100（100点満点に換算）
```

※重み付けは運用後に調整できる設計にする

### 診断結果の基準

**総合点（100点満点）**

| スコア | 判定 | 表示メッセージ |
|--------|------|--------------|
| 80点以上 | 優秀 | チームの状態は非常に良好です |
| 60〜79点 | 良好 | チームの状態は概ね良好です |
| 40〜59点 | 要注意 | いくつかの課題があります |
| 39点以下 | 危険 | チームの状態に深刻な課題があります |

**各軸スコア（6点満点）**

| スコア | 判定 |
|--------|------|
| 5.0以上 | 強み |
| 3.5〜4.9 | 普通 |
| 3.4以下 | 弱み・要改善 |

## 12. その他仕様

### 自由記述のAI分析
- Claude APIを使用してサマリーを自動生成
- 個人名などは送信前にマスキング処理を行う
- 原文はアドミンのみ閲覧可能・メンバーにはサマリーのみ表示
- 利用規約にAI分析の旨を明記して同意を取る

### アンケート配信方法
- メール通知（招待メールと同様に配信）
- URL共有（SlackなどでURLをシェア）の両方に対応

### データ保持・削除ポリシー
- データは無期限で保持
- アドミンが手動で削除できる
