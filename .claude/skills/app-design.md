# スキル: アプリ自立設計（app-design）

## このスキルの使い方
「〇〇アプリを設計して」と依頼するだけで、このプロセスを自立的に実行する。

---

## 自立型デザインプロセス

### STEP 1. 課題特定（市場リサーチ）

**必須ルール：ハルシネーション禁止**
- WebSearch / WebFetch ツールで実際に検索・取得した情報のみ使用する
- 情報が見つからない場合は「情報なし」と明記し、推測で補わない

調査内容：
- 市場規模・トレンド（検索して確認）
- 競合サービス（実在するものを列挙）
- ユーザーの未解決課題（レビュー・口コミ・記事などから）

アウトプット：
- 課題ステートメント（1〜2文）
- 競合比較表（実在サービスのみ）

---

### STEP 2. ユーザー定義（ペルソナ作成）

**メインペルソナを必ず1体作成する。**

サブペルソナの判断基準（該当する場合は追加）：
- B2Bサービスで「購買者」と「利用者」が異なる
- 管理者・承認者など複数ロールが存在する
- 意思決定者と実際の操作者が異なる

ペルソナ記載項目：
```
名前：（架空でOK）
年齢・職業：
利用シーン：
抱えている課題：
行動パターン・習慣：
技術リテラシー：
```

アウトプット：
- メインペルソナカード
- サブペルソナカード（必要な場合）

---

### STEP 3. 要件定義

- 機能要件：ペルソナの課題を解決するために必要な機能を列挙
- 非機能要件：パフォーマンス・セキュリティ・スケーラビリティなど
- スコープ外：このバージョンでは作らないものを明示

---

### STEP 4. 技術スタック選定

このワークスペースの既存スタックを優先して提案：
- Python / Flask（分析・レポート系）
- TypeScript / React / Vite / TailwindCSS（UI系）

既存スタックで対応できない場合は理由とともに代替案を提示する。

---

### STEP 5. 実装計画

- フォルダ構成（`app_` から始まるフォルダ名）
- 主要ファイル一覧
- 開発の優先順位（フェーズ分け）

---

### STEP 6. コンセプト検証

視点：「このアプリは本当にペルソナの課題を解決できるか？」

チェック項目：
- [ ] 課題とソリューションが対応しているか
- [ ] 競合との差別化ポイントがあるか
- [ ] 各ペルソナにとって価値があるか
- [ ] ビジネスとして成立するか（必要に応じて）

→ 問題があれば STEP 2〜5 に戻って修正する

---

### STEP 7. デザイン検証

視点：「アーキテクチャ・技術選定は適切か？」

チェック項目：
- [ ] 技術スタックは要件を満たせるか
- [ ] 複数ペルソナのユースケースに対応できる設計か
- [ ] 将来の拡張性はあるか
- [ ] 開発コストは現実的か

→ 問題があれば STEP 4〜5 に戻って修正する

---

### STEP 8. ユーザビリティ検証

視点：「各ペルソナが迷わず使えるか？」

チェック項目：
- [ ] メインペルソナが主要機能に迷わずたどり着けるか
- [ ] サブペルソナ（存在する場合）のフローは整理されているか
- [ ] エラー時のフィードバックはわかりやすいか
- [ ] 技術リテラシーが低いペルソナでも使えるか

→ 問題があれば STEP 3〜5 に戻って修正する

---

### STEP 9. ドキュメント保存

最終版を自動保存する：
- 保存先：`docs/requirements/YYYY-MM-DD_アプリ名_設計.md`
- 各プロジェクト内の場合：`Project/app_*/docs/requirements/`

保存内容：
- 課題ステートメント
- ペルソナカード（全員分）
- 要件定義
- 技術スタック
- 実装計画
- 検証結果サマリー（各検証で発見した問題と対処）

---

## 注意事項

- **ハルシネーション禁止**：市場情報・競合情報は必ず検索して取得する。不明な場合は「不明」と記載する。
- **ペルソナ数はサービスに応じて判断**：増やしすぎると設計がブレるため、3体以内を目安にする。
- **検証で問題が出たら必ず戻る**：「問題なし」で通過させない。
